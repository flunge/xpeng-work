# 正式文档撰写与审阅规范

向公司/管理层提交的文档需体现负责人的判断力与风格，落笔或审阅时逐条核对。

## 一、内容红线

1. **不写晋升**：不主动建议或暗示某人晋升（晋升是 1:1 / 管理层口头话题）。
2. **人名用 @ 或 `<cite>`，不裸写**：结构性事实（某人离职/加入）保留并加 @；评价人、细碎分工不写。判断标准——文中每个名字是否都用了 @ 格式。
3. **精准 scope**：只写本组职责范围内的事；写之前确认边界，别把其他组/平台组的业务算进来。
4. **信息来源有边界**：群聊表扬 ≠ 正式素材；日会细节 ≠ 公司级内容。

## 二、不要 AI 腔

落笔后自查：「这段话像我向上级当面汇报时会说的吗？像人话吗？」典型 AI 写法须改掉：
- 在替读者下评价（「已具备独立交付能力」「比外部招聘更能激励团队」）
- 套话（「形成可复用的范式」「持续放大…的价值」「AI Native 的首次尝试」）
- 口号式对仗（「从 X 推到 Y」结构过整）
- 暗示晋升、把群聊八卦当素材

## 三、写短

表格单元格砍到要点，文字块只说判断与结论、不铺陈过程；默认规则（如打卡制度）不解释。能删就删。

## 四、发布前专项检查（对外文档必做）

```
□ ASR 术语扫描：全文 grep 易错音译词，结合本组语境校正——
    不要照搬会议文字记录的原词，要理解它在我们语境下到底指什么。
    已知映射：黑友→HIL、gspend/gspan/gisplan→gsplat、克劳森/cloudsion→CLIP-IQA、
              NV fixer→NVFixer、克隆界面→clone 流程（按上下文判断）
□ 数字单位一致：百分比/km/台数/复现率全文统一，且与源数据一致；
    同一指标在正文与总结/反思段不得出现两个不同口径的数（如 67% vs 87% 需说明来源或对齐）
□ 人名一致：姓名全程统一，不混用昵称/英文缩写/全名
□ 新名词有背景：首次出现的技术名词（UCP/CCES/PAT/CLIP-IQA/Fixer/difix 等）附一句话定语
□ 红黄灯上限：单板块最多 1🔴+1🟡（GIC 规则）
```

## 五、落笔前三问

这像我说的话吗？这是事实还是评价？需要写这么长吗？

## 六、禁止元叙述

文档本身就是某种格式（PDCA/OKR/述职/周报）时，正文中**不要自我引用格式名称**。例如：
- ❌ "从 PDCA 角度看……"（文档本身就是 PDCA，读者知道）
- ❌ "在本述职中我们……"
- ✅ 直接写内容，不提文档类型
