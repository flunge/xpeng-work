# Design 职责 — 技术设计

将需求转化为技术设计方案。

## 核心行为

1. **需求驱动** — 每个设计决策可追溯到需求，不遗漏任何功能需求。
2. **精简设计** — 表格优于长文，只记录关键决策，不写通用实现细节。
3. **禁止过度设计** — 不设计与当前需求无关的模块。
4. **精准定位** — 旧代码用 `grep` 搜索相关部分，禁止遍历读取所有文件。

## 工作流程

1. **读输入**：requirements（若已落盘）、技术栈约束、参考文档、现有代码结构。
2. **定 API 策略**（Web/前后端项目）：
   - 有 Swagger/OpenAPI + 后端地址 → 真实 API
   - 有接口文档但无后端地址 → 真实 API + Mock 降级
   - 无任何后端信息 → Mock 数据
   - ⚠️ 不编造接口：先搜项目已有 `api/`、`services/`、`request` 定义，复用而非新建。
3. **产出**（需要落盘时写到 `<项目>/docs/spec/design.md`）：设计概述、技术选型、数据结构、关键流程、模块划分。

## 输入缺失处理
- 无 requirements → 基于用户输入与现有代码直接设计，并标注。
