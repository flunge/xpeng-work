# 铁律 · 溯源与采集（sourcing）

> 单一事实源。所有"报告/修改/更新记忆"前的信息获取纪律都在此，别处只引用。

## 1. 溯源先行（第一铁律，所有报告/所有修改适用）
永远**先溯源、收集足够内容，再动笔**。禁止在"已有文字表面"上删改（那是缩写、越改越看不懂）。正确动作：先 fetch 源文档 / 读 ledger / 翻 chat-log，把"这词、这步在解决什么问题、因果链是什么"吃透，收集到丰富上下文后再写，用简单语句讲清 **痛点→思路→做法**。判据：改任何一句前问"我读过它的出处吗、知道它在解决什么问题吗"，没有就先溯源。

## 2. 溯源=吃透全貌，不是点状核对
致命陷阱：先预设要写哪几句、再逐句找出处，核对完就以为够了——这样只能在自划的小池子里挑重点、必然挑不准。**判断"什么是重点"必须先看到全部候选。**
- 正确动作：写每个 Topic/项目前先做**全貌摸底**——读完整 ledger（不是 grep 关键词）+ 最近数次日报里这条线的连续进展 + 上下游/业务方诉求 + 汇报对象关注点，先产出"在解决什么→演进到哪→当前卡点→下一步"的脉络，再从脉络里选重点。
- 充分性判据（写前必须能答）：领导追问"这条线整体走到哪？下一步？最大风险？这个数字为什么重要？"——答得上才动笔，答不上=信息不足，回去补。

## 3. 逐条回源自查（写完发出前，动手查、不是自问）
把报告拆成逐条 bullet 编号 → 把本周期全部源头（两份日报 + 每场纪要/逐字稿 + chat-log 本周段）grep 进临时语料 → **每条关键事实（数字/状态/动作/时间）拿去源语料搜**：
- ✅ 搜得到且一致 → 保留
- ⚠️ 搜得到但我改了述（数字/修饰词/确定性）→ 改回原述
- ❌ 搜不到 → 是旧 ledger/记忆/非正式来源 → 删
把逐条核对结论（留/改/删+源）列出，不只说"校验通过"。

## 3.4 改文件名必带扩展名（2026-07-16 沉淀）
`lark-cli drive files patch --data '{"new_title":"X"}'` 改云盘文件（type=file）标题时，**new_title 必须带原扩展名**（`.pdf`/`.jpeg`/`.xlsx`…）。漏了扩展名→飞书失去文件类型识别→预览显示"文件已损坏/无法读取"（改名其实成功了，inspect 能查到正确 title，是扩展名缺失导致的预览异常，不是文件坏）。批量改名后务必核对扩展名。

## 3.5 读独立 PDF/图片文件（2026-07-16 沉淀）
飞书云盘里的 PDF / 扫描件不能直接 Read（guard 拦二进制，且 PDF 文本层常为空=扫描件）。流程：`lark-cli drive +download --file-token <tok> --output ./x.pdf --overwrite`（`--output` 必须相对 cwd）→ 有文本层的先 `pypdf` 提文本；**扫描件/图片型**用 `pymupdf` 转 PNG（`fitz.open(pdf)[0].get_pixmap(dpi=N).save('x.png')`）再 Image 模式 Read。⚠️ **Read 图片有 512KB 上限**——大图调低 dpi（110→70→40）直到 <512KB。批量识别未知文件名（如解压丢名的 `garbled_*`）就靠这个：一张张读图确认真实内容再改名/归类。（本次用它识别 15 个 garbled 文件：引进人才申请表/学位证/职称证/户口卡/社保记录等，还发现 MD5 相同的重复件。）

## 4. 🔴 日报贴图必读（2026-07-09）
作战表 W## 每日进展 cell 里大家贴的 `<img>`（性能/latency 表、metric ETA 表、车型×场景矩阵、case review 结果、效率统计截图）**必须逐张下载读全**：从 W## 段 grep 全部 `<img src>` → `docs +media-download --token <src> --output x.png` → Image 模式读。**alt 文本只是 AI 粗描述、常漏精确数字**（如 latency=130.2ms/trt=86.7ms、10 车型各 metric 场景数、review 准确率 86.2%），图里才有。漏读图 = 采集未完成。
- **🔴 机器闸（2026-07-13，跑不过发不出）**：读一张作战表贴图就把它的 src token 追加进 `scripts/.tmp/.img-read-log`（一行一个）；报告发布前跑 `python3 scripts/check_report.py <report_token> --audience boss|xianming --wartable SBUYwm8Lri9aJ6kmexFcBAuGnlh --weeks W28[,W29]`，它打印「作战表贴图 已读 N/总数」并把每张未读图列为拦截项（exit 2）。未读清零才放行——用闸兜住"逐日读图"，别靠自觉。

## 5. 🔴 数字口径（2026-07-09）
① 分清「总累计(含历史基线)」vs「本月/本周新增」——RC"新数据累计 946km"是本月新增，"累计破 3500km"是总累计（月初基线 ~2500 + 本月新增）。② 「X→Y」区间目标：终点是 Y、起点 X 是基线，**别把起点当月底目标**（RC 月度 2500→5000km，月底目标 5000km 不是 2500）。写每个里程/数量前先判定它是累计还是增量、是起点还是终点。
③ **数字不改述**：源说 1300+ 只能写 1300+，不能写"近 1500/约 1500"；旧周数据无新进展只能进"现状"标"此前已验证"，禁止冒充"进展"。

## 6. 🔴 不懂就问，严禁脑补
遇到以下立即向用户确认：不认识的缩写/术语（UCP/CPFS/Seal→SIL 等）、不确定项目边界、不认识的人、不确定数字、两源矛盾。

## 7. 三源配对
进展须 **日报(作战表 Q2=`PDJ2wKHn`/Q3=`SBUYwm8L`) + 日会(智能纪要) + 逐字稿** 三源配对，同日多源分列/综合理解。

## 8. 来源分级（能否作"进展"）
- ✅ 可作进展：核心日会/组内日会/每日例会纪要+逐字稿的正式汇报、作战表本周日报列。
- ❌ 仅参考（不进进展）：茶水间、idea 会、头脑风暴、群里随口评论/口头预估（带"预估/后续/测试中/计划"字样）。判据：正式场合产出的可交付/可验证结果吗？

## 9. 读纪要必读逐字稿（原 meeting-rules 并入）
每次读会议纪要**必须同时读文字记录(逐字稿)**——智能纪要是 AI 概括、丢失个体声音；逐字稿才有每句谁说的、语气、真实关注点。VC Assistant 推送的是智能纪要、内含逐字稿链接，读完主动找。核对标准："老板在这会上具体说了哪几句话？"答不上=没读逐字稿。（教训：6/10 只读智能纪要，漏掉高炳涛唯一硬要求"Bazel 改 CMake"。）

## 10. 跨会议重复信号（4 找，归并不孤立）
遍历逐字稿别当孤立文档逐篇处理，要按**发言人×主题跨周归并**：
- 同一人 ≥2 次会提同一阻塞/依赖 → 标🔴高优风险，在项目 ledger+人物文件风险栏写"连续 N 周汇报"。
- 核心汇报人有固定节奏(李坤周会) → 对比本周与上周同主题发言，标出"未解卡点"。
- 某外部组被多次提为依赖 → 在**该组负责人人物文件**写"核心阻塞依赖"节(如夏志勋 metric)。
- 静默检测：上次报告出现过的项目/风险本周完全没出现=危险信号，追问一次。
（教训：李坤 5/27→6/3→6/10 连续三周"等志勋 metric ready"被漏记为孤立事件。）

## 11. 🔴 嵌套文档按 revision diff 抽增量（2026-07-13，读文档标准做法）
读项目嵌套文档时**别每次全文重读**——用版本 diff 精准抽窗口内的实际更新：
1. `lark-cli docs +history-list --doc <token>` 拿版本历史（每条含 `edit_time` + `revision_id` + `editor_ids`；默认返回近 20 版，更早用 `--page-size 1-20` + `--page-token` 翻页）。
2. 取「窗口起点前最后一版」为基线 rev_old、「最新」为 rev_new；`docs +fetch --doc <token> --revision-id <rev> --doc-format markdown` 各取一版。
3. 行级 diff（新增/删除行）＝窗口内的实际更新；`editor_ids`（open_id）映射到人，知道谁改了什么。
- **优点**：精准（只看真改动）、不漏更新、省 token。是【更新记忆】/周报对各项目嵌套文档抽增量的默认做法。
- **注意**：超出保留期的老版本可能取不到（history-list 拉不到就退回全文读）；要精确到"某句改成某句"再做句级逐行对比。
- 实测（G6I4w 7/9→7/13）：diff 直接抽出"五阶段大批量生产/单卡4任务/共用h265解码/11.8min benchmark 算式"，无需重读全文。
