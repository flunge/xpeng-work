# 命令：【更新记忆】

> **唯一的记忆同步命令**（2026-07-09 统一：原【团队工作更新】【零·启动同步】已并入本命令）。用户在聊天窗口发 **【更新记忆】** 时执行。
> 核心原则：**理解 → 协调 → 修订，不是粗暴增量**；日报进展必须系统同步进 memory 的项目/人员/团队 ledger + chat-log，不能只靠会议纪要、不能"用时现读"。

## 一、更新原则：理解 → 协调 → 修订
每条新信息先**读懂现有记忆**再决定动作：
- **新增**：记忆没有的，按日期/分类加到对应位置。
- **修订**：新信息推翻/细化旧记忆 → 直接改旧条目（指标 92%→95%、方案 A→B）。
- **解除/收敛**：记忆里的风险/阻塞/待办若已解决 → **标记解除或删除**，不让过期风险一直挂着（如某链路 OOM 已修 → 更新为已解决+日期）。
- **关联**：新进展与项目 ledger/人物画像/洞察交叉呼应，同一事实不在多处矛盾。
- 目标：记忆**始终反映当前真实状态**，非只增不改的流水账。

## 二、时间窗口（自维护）
1. **取起点**：读 `team/memory/.last-memory-update`（为空则用更早的安全值）。
2. **取终点**：当前系统时间。
3. **算区间**：起点→终点即本次窗口；**先报出窗口再干活**。
4. **收尾必做**：把当前时间点写回 `team/memory/.last-memory-update`。

## 五、写入（三线 + chat-log + 索引）

- 每条标来源；遵循 `rules/writing.md` 安全编辑。

```

Phase 3 聊天上下文：reply_to 读父消息、消息内链接读全、搜到消息读 chat 前后文
Phase 4 理解现有记忆 → 协调修订(新增/修订/解除/关联)写三线 + chat-log
Phase 5 索引维护 + 收尾(.last-memory-update 写回)
Phase 6 自检：每条能指到源？涉及的 project/people/chat-log 都更新了？读源还是脑补？过期风险解除了？
```
