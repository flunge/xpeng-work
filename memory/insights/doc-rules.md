## 正式文档撰写规范

**Why:** 用户在审阅 H2人力规划文档时多次纠正。这类文档提交给公司，需体现P8组长的判断力和风格。

### 一、不写晋升

不主动建议或暗示某人晋升。晋升是1:1/管理层口头话题，不写入正式文档。

### 二、人名：用 @格式，不裸写

- **该写**：结构性事实（杜思聪离职、某某加入），用 `@人名` 或 `<cite>` 标签
- **不该写**：裸写人名（"杨星昊提案"→"@杨星昊 提案"）、评价人、细碎分工
- **判断标准**：文档中出现的每个名字，是否用了 @ 格式？

### 三、精准 scope

算法组 = **闭环仿真**（SIL/HIL/生产链路/算法预研）。具体方向：3DGS场景重建、Fixer、Agent、World Model预研、AVM鱼眼、场景泛化、CLIP-IQA。

**不属于算法组 scope**（写之前查 [[team-scope]]）：
- 海外、机器人、RT、汇天X3（其他组的业务线）
- 传感器仿真、交通流建模（不是我们做的）
- 大规模并行仿真调度（平台组的事）

### 四、不要 AI 腔

以下是从本次文档中被揪出来的典型 AI 写法，以后不准出现：

| 错误写法 | 问题 |
|----------|------|
| "已具备独立交付能力，处于快速成长期" | 在评价人 |
| "AI Native角色的首次尝试" | 不像人话 |
| "形成可复用的Agent开发范式" | AI 套话 |
| "比外部招聘更能激励团队" | AI 在替你下判断 |
| "OKR Owner集中在个位数" | AI 在编词 |
| "积累晋升证据" | 暗示晋升 + AI腔 |
| "亮点：XX获部门负责人公开表扬" | 群聊八卦当正式文档素材 |
| "全员上下班打卡，不存在远程办公情况" | 默认规则不用解释 |

**自查方法**：这段话像我在和高炳涛汇报时会说的吗？像人话吗？

### 五、写短

- 表格单元格：砍到要点，不要大段叙述。C3从400字砍到180字还不够
- 文字块：只说判断和结论，不铺陈过程
- 能删的就删，HR知道默认规则

### 六、信息来源要有边界

- 群聊里的表扬 ≠ 正式文档素材
- 日会纪要的细节 ≠ 公司级文档内容
- 从 [[chat-insights]] 提取的信息需要判断是否适合写入正式文档

**How to apply:** 落笔前问三问：这像我说的话吗？这是事实还是评价？需要写这么长吗？
