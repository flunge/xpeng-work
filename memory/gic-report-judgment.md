## 角色定位

李坤是P8组长，汇报对象是仿真部三级部门，**不是仿真部负责人**。不能从部门负责人的视角汇报组织架构调整等大组层面的事。

## "进展"的定义校准

进展≠做成了什么。以下都值得汇报：

1. **有明显进展的事**——产出、数据、里程碑
2. **投入人力较多的事**——即使没有明显产出，人力投入本身就是信号
3. **遇到的问题和弯路**——这也是信息量，而且对上级判断风险很重要

**Why:** 用户指出我之前的判断偏向"做成了什么"，忽略了"在做什么但遇到困难"也是汇报的重要组成。GIC的读者需要知道团队的时间花在哪里，而不只是成功的部分。

**How to apply:** 读双周报文档时，先标出所有投入人力的条目，再按"进展/投入/问题"三个维度分类，而不是只挑"做得漂亮的"。IT部署延迟、OLM根因修复中这类"问题项"也是有效进展——说明团队在推进且有风险需要关注。

## 话题必须挂在大 Scope 下

刘先明（GIC老板）记不住组里细粒度的事项。他脑子里只有几个大桶：

- **闭环仿真 SIL/HIL**（场景生产、运行效率、评测体系、HIL台架）
- **3DGS 生产**（数据预处理、训练、模型交付）
- **Agent 技术创新**（AI看复现、代码审计、智能工具）

**Why:** 用户指出如果我不主动把每条进展归类到这些大 scope 下，信息到刘先明那里就散掉了——他听完只记得一堆细节，记不住"李坤组这周在闭环仿真上做了什么"。归类不是偷懒，是帮上级建立稳定的认知框架。

**How to apply:** 
1. 每条进展必须归类到大 scope 下再呈现，例如"极速模式 2.6x 提速"→「闭环仿真 - 场景生产效率」；"Metric 36→12 Behavior"→「闭环仿真 - 评测体系」；"AI看复现 Review"→「Agent 技术创新」
2. 先给大 scope 的结论，再展开细节。例：「闭环仿真方向，这周效率大幅提升——极速模式 2.6x 验证通过，UCP 下周上线」
3. 如果一个进展挂不上任何大 scope，要么是新方向值得单列，要么就不够重要
4. 不要为了凑齐所有桶而强行总结每个方向——某个方向本轮没有重点进展就略过，不要硬塞，更不要编造"暂无突破"之类的评价
5. **禁止编造定性判断**：所有结论必须来自源文档原文。不能说文档没写的评价（如"暂无重大突破"），不能自己给进展打分或加形容词，不能脑补"问题/风险"里没提到的东西

## 元规则：每次聊完汇报逻辑后做两件事

1. **核查**：上述所有规则是否已在 agent 记忆/配置中落地，没有遗漏
2. **产出**：根据最新规则生成新文档（分析、报告、示例），作为规则落地的具体产出

**Why:** 用户不希望讨论完规则就结束了——讨论的终点是"规则已落地 + 有可见产出"，否则下次还会犯同样的错。
