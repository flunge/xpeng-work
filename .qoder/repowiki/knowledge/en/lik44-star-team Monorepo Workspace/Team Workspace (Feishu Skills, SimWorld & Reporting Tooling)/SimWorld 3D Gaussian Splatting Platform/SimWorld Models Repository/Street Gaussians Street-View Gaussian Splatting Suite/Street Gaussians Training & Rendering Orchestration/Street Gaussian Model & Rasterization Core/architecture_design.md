Two-layer hierarchy inside the module:
- Base layer: `gaussian_model.py` defines `GaussianModel(nn.Module)`, the canonical 3D-GS primitive holding `_xyz/_features_dc/_features_rest/_scaling/_rotation/_opacity/_semantic` as `nn.Parameter`s with SH activation helpers, PLY I/O, densification (clone/split/prune), two-phase Adam optimizers, and an optional per-view `AppearanceNetwork` + embedding. Specialized subclasses extend it: `gaussian_model_bkgd.py`, `gaussian_model_grd.py`, `gaussian_model_sky.py`, and `gaussian_model_actor.py` (adds time-varying Fourier features over frames, per-object bounding box, and class-scoped semantic).
- Composition layer: `street_gaussian_model.py`'s `StreetGaussianModel` dynamically instantiates one of each sub-model based on config flags (`include_background|ground|sky|obj|quasi_static_obj`) and a `bidict` mapping model names to contiguous index ranges; `parse_camera` / `parse_camera_dynamic_objects` build a per-frame scene graph, concatenate visible Gaussians into unified `get_xyz/get_scaling/get_rotation/get_features/get_semantic/get_opacity` properties, and optionally apply pose correction (`camera_pose.PoseCorrection`) and color correction (`color_correction.ColorCorrection`). Actor poses are interpolated from tracklets via `actor_pose.ActorPose`; sky can be rendered as a `sky_cubemap.SkyCubeMap`.
- Rendering layer: `gaussian_renderer.GaussianRenderer` wraps `diff_gaussian_rasterization.GaussianRasterizer` for the base model; `street_gaussian_renderer.StreetGaussianRenderer` dispatches between `render_kernel_diff_gaus_raster` and `render_kernel_gsplat` depending on `cfg.train.use_gsplat`, composes foreground/background/ground/object passes, blends sky cubemap, applies color correction, and returns rgb/acc/depth/viewspace_points/radii plus optional normals/semantic features.
- Orchestration: `scene.Scene` owns a `Dataset` and either constructs a `GaussianModel`/`StreetGaussianModel` from point clouds (train mode) or loads a checkpoint iteration (resume/render mode).
Dependency direction is strictly upward: renderers depend on models, `Scene` depends on both, and all files import shared utils from `lib.utils.*` and global `lib.config.cfg`. The `network_gui.py` file provides a separate Gradio-based visualization entry point not used by the training loop.
