Three-layer hierarchy rooted in `base/`:
- `SceneDataset` (abstract) defines the dataset contract (`build_data_source`, `build_split_wrapper`, `split_train_test`) plus train/test split bookkeeping; `DrivingDataset` is the concrete implementation that wires together a pixel source, an optional LiDAR source, a `NovelViewManager`, and `IterableSplitWrapper`s for train/full/test views.
- `ScenePixelSource` / `CameraData` abstract per-camera image loading, calibration, mask handling and ray construction; each dataset folder (`waymo/`, `kitti/`, `nuscenes/`, `argoverse/`, `pandaset/`, `nuplan/`, `xpeng/`) provides a subclass of `ScenePixelSource` plus a matching `WaymoCameraData`-style `CameraData` subclass implementing `load_calibrations`, `create_all_filelist`, and object/instance parsing.
- `SceneLidarSource` abstracts LiDAR point clouds (origins/directions/ranges, world poses, visibility masks, colors); each dataset supplies its own loader.
- Dataset selection is runtime-dispatched via `import_str(self.data_cfg.pixel_source.type)` / `self.data_cfg.lidar_source.type`, so adding a new dataset only requires a new sub-package implementing the two base classes.
- Cross-cutting helpers live in `tools/` (masks, SMPL extraction, multiprocessing), `dataset_meta.py` (DATASETS_CONFIG mapping dataset name → camera list/names), `preprocess.py`, and `novel_view_manager.py` (caching pre-rendered novel views). The module depends on sibling packages `..utils.camera`, `..utils.geometry`, `..utils.visualization` but has no internal cross-imports between dataset sub-packages.
