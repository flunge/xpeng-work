- `main.py` is the single entry point: it iterates over configs returned by `settings.config.get_config_list()`, calls `make_case_specific_settings` to resolve clip-specific overrides, then branches on `cfg.steps_controller.source` (`lidar` vs `vision`) to run two-stage pipelines.
- `pipelines.py` defines four orchestrators — `pipeline_m1_lidar_cpu/gpu` and `pipeline_vision_cpu/gpu` — each instantiating one `*Processor` class per step (Json/Pose/Range → Img → Opt → Lidar/Seg/Mask → COLMAP → Point/Densify → GroundSurfel → Depth → G3R/EvoSplat/SAM3D/Mvsnet/PcdFusion/TrafficLight). Steps are gated by boolean flags under `cfg.steps_controller.*`, so any stage can be skipped without code changes.
- Child modules are consumed exclusively through thin wrapper processors at this level (`ground_processor`, `mvsnet_processor`, `sam3d_processor`, `gsm_processor`, `opt_processor`, etc.); they never import each other directly. Cross-module data exchange happens via files written into the per-clip directory tree (images, masks, segs, colmap DB, PLY, TSDF, surfels), not in-memory objects.
- Shared state is centralized in `settings/config.py` + `settings/globals.py` (YACS config merged from `configs/*.yaml`) and `utils/*` helpers (COLMAP parsing, projection, OSS I/O, depth/point utilities); every processor reads/writes through these instead of hard-coded paths.
