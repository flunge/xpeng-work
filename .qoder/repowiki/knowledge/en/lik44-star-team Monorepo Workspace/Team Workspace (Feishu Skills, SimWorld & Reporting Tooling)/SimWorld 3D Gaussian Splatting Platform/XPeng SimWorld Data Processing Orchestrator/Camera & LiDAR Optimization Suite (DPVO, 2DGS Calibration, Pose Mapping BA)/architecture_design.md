Three sibling sub-packages share a common Docker image (`optimization/Dockerfile`) built on an internal `fuyao` base with CUDA 12.6 and PyTorch 2.7:

- `camopt/` — self-contained fork of the DPVO monocular SLAM stack. `dpvo/dpvo.py` exposes the `DPVO` class which orchestrates `altcorr` (CUDA correlation), `fastba` (block-sparse CUDA bundle adjustment), `lietorch` (SE3/SO3 Lie-group ops), `patchgraph`, `loop_closure`, and `sam2` segmentation. Subpackages `DPRetrieval` and `DPViewer` are pybind11+CMake extensions for dense-pairwise retrieval and GPU-accelerated visualization respectively; they ship their own vendored `pybind11/` trees and `setup.py`+`CMakeLists.txt` build manifests.
- `lidaropt/lidar2cam_opt/` — extrinsic calibration pipeline driven by `pipeline_lidar2cam.sh`. It chains: data prep (`process/gen_global_pcd.py`), SuperPoint+SuperGlue feature matching (via `submodules/Python-VO`), 2DGS scene reconstruction (`geometry/main.py` using `diff-surfel-rasterization`), then differentiable pose/gaussian optimization in `calibration/{pose_model,gaussian_model}.py` backed by Adam + ExponentialLR. The script writes a `res.json` containing the optimized delta transform.
- `posemapping/` — pure-Python BA front-end (`bundle_adjustment.py::BundleAdjustment`) that serializes a `Vehicle` graph (poses, cameras, lidars, landmarks, matches) into ctypes structs and calls the prebuilt `ceres_ba/libceres_ba.so`; bidirectional converters live in `ctypes_utilis.py`. Supporting modules (`sensor.py`, `vehicle.py`, `keypoint.py`, `superglue_match.py`, `triangulation.py`) model the optimization problem before it is handed off.

Dependency direction is one-way: `lidaropt/pipeline_lidar2cam.sh` is the top-level orchestrator; `camopt` and `posemapping` are independent libraries invoked from scripts or notebooks. All heavy computation is delegated to CUDA kernels or the external Ceres binary.
