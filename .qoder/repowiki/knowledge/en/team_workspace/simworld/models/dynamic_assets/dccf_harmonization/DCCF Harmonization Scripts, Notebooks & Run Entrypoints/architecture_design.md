Three sibling top-level folders form a thin orchestration layer over the upstream `iharm` package:
- `scripts/`: CLI entry points. `evaluate_upsample_refiner.py`, `evaluate_model_rawsize.py`, `predict_interactive.py`, `harmonization_evaluator.py` (which exposes `HarmonizationEvaluator` + `create_harmonization_evaluator`) and `cam_image_harmonize.py` (the integration bridge used by the simulator) all import from `iharm.*` (`mconfigs`, `data.hdataset`, `inference.predictor_upsample_hsl*`, `evaluation`, `metrics`, `utils.exp/log`). Each script rewrites `sys.path` to reach the parent `DCCF/` directory so it can locate its own `config_test_*.yml` files.
- `notebooks/`: one-off data-prep utilities (`resize_dataset.py`, `_1024_1024.py`, `_2048_2048.py`) that copy an IHARM-style dataset tree under `composite_images/masks/real_images` while resizing images to a fixed max dimension.
- `runs/*.sh`: executable training/test pipelines. They call `train.py` (outside this scope) with model configs under `models/fixed256/...`, then chain into the `scripts/*` evaluators; multi-GPU runs use `torch.distributed.launch`. Baseline scripts exercise alternative upsample methods (BGU, bilinear, GF, rawsize) via `evaluate_model_rawsize.py`.
Dependency direction is strictly outward: this module depends on `iharm` and on sibling `models/fixed256/` config files, but nothing inside this folder imports back into `iharm` — it only consumes its public inference API.
