Two-layer design:
- Runtime layer (`agent_service_main.py`): `AgentService` loads a YAML sim config, parses ego poses from `ego_frame_poses`, then delegates lifecycle to `AgentManager`. Each frame calls `on_step(timestamp)` which iterates all `DynamicAgentVehicle`s (managed via the C++-backed `AgentDataField`) and every active action delegate.
- Action/Trigger layer (`actions/`): concrete `ActionBaseDelegate` subclasses (`ChangeLaneDelegate`, `SpeedAccelDelegate`, `GenerateAgentDelegate`, `RemoveAgentDelegate`, `ModifyDriverDelegate`) are registered in `AgentManager.create_actions`; each holds an ordered list of `ActionBaseTrigger` subclasses (`EgoMileageTrigger`, `RelativeDistTrigger`, `EgoTimeTrigger`, …) whose `alive()` + `is_trigger_activated()` gates the transition from WAITING→RUNNING→FINISHED.
- Agent layer (`agents/`): `Agent` → `DynamicAgent` → `DynamicAgentVehicle` hierarchy; `DynamicAgent` implements a simple longitudinal planner (`longitudinal_move_planning`, `go_straightly`, `run_in_acc_mode`) driven by `driver_param` (max_speed, max_acc, target_speed) and exposes `to_sf_dynamic_obj` / `to_sf_static_obj` for sensor-fusion export.
- Orchestration layer (`pipeline.py`): multi-threaded worker pool (`THREAD_COUNT=5`) over `CLIP_3DGS_CONFIGS`; each task downloads a clip's `3dgs_model.tgz` from OSS, backs up `configs/config_sim.yaml`, runs `generate_new_config` to rewrite it with injected objects, repackages via `upload_model.create_tgz_archive`, and uploads back to `sim_engine/artificially_created_scenes/<clip_id>/trained_model_<ts>/`.
- Config layer (`configs/`): CSV manifests (`clips_config*.csv`, `dynamic_obj_configs.csv`) plus per-scenario JSON controllers under `agent_service_configs/` drive both pipeline dispatch and runtime agent/action creation. `configs.config` is the single source of truth for `CLIP_3DGS_CONFIGS`, `AGENT_SERVICE_CONFIGS`, `DYNAMIC_OBJECTS_CONFIGS`, `INCLUDE_DIRS`, `INCLUDE_FILES`.
