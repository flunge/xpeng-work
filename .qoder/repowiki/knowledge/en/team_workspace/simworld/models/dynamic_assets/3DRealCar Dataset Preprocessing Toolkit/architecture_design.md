The module is a self-contained toolkit (`three_D_RealCar_Toolkit/`) whose entry points are Bash orchestrators under `data_preprocess/bash/`. `pipeline.sh` is the single CLI dispatcher: it validates arguments, resolves the current processing stage by checking `.processed` / `.segmentation` / `.dataset` marker files (making each step idempotent), then invokes one Python script from `utils/toolkit/`. The Python layer follows a clear separation:
- `entrances/dataset_adaptor.py` adapts heterogeneous inputs (3dscanner, Colmap, ARKit) into a unified Colmap layout via subclasses in `utils/adaptor/dataset.py`.
- `utils/toolkit/core.py` defines the abstract `LiGSToolkit` base class that loads Colmap data, links images/alphas/arkit frames, writes PLY + Colmap binary outputs and touches `.processed`; concrete stages (`extract_foreground_pcd`, `standarize_coordinates`, `rescale_colmap`, `extract_segmentation`, …) subclass it and implement `run()`.
- Stage-specific helpers live alongside their callers in `utils/toolkit/` (e.g. `depth_extraction/colmap2mvsnet.py`, `vismvsnet_filter.py`).
- Shared I/O, logging, visualization and utility modules sit under `utils/{io,logging,visual,utility}` and are imported by every stage.
- Per-experiment hyperparameters come from YAMLs under `resources/configs/` (e.g. `demo.yaml`, `vehicle_config.yaml`, `vru_config.yaml`); `pcd_standard.txt` holds manual overrides for coordinate standardization.
Dependency direction is strictly top-down: bash → entrance → toolkit core → stage scripts → shared utils; there is no cross-import between sibling stage scripts.
