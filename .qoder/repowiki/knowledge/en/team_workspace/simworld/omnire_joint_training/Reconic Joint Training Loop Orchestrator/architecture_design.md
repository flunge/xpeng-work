The module follows a classic hook-based training-loop pattern centered on `TrainingLoopHelper` (`training_loop_helper.py`). The helper owns lifecycle: config loading (OmegaConf merge of base + dataset YAML + CLI opts), dataset/DataLoader construction, `recon_trainer` instantiation via `import_str`, checkpoint resume from `training_metadata.json`, the main `train()` step loop over `self.all_iters`, per-step hooks (`run_before_train_step`, `forward_step`, `backward_step`, `run_after_train_step`, `run_after_step_finished`), metric logging to `metrics.json`, periodic error-map caching, and wandb sync. Three concrete subclasses override only the forward/backward/step-finished hooks:
- `ReconTrainingLoop`: vanilla single-view reconstruction.
- `XpengTrainingLoop`: adds a `Recorder` in `run_after_step_finished` for extra telemetry.
- `GenerativeReconTrainingLoop`: implements two-phase joint training driven by `joint_training_cfg` (`start_engine_infer_at`, `stop_engine_train_at`, `iterations_per_shift`, `max_shift`). It builds a dynamically imported generative scheduler/engine pair, runs recon forward on real images plus optional synthesized novel views, pushes render↔GT pairs into the scheduler's queue, and uses a daemon `_store_worker` thread to persist generated novel-view data to disk while respecting shift bounds derived from `pixel_source.max_range_info`. Checkpointing is split between the recon model (`trained_model/checkpoint_*.pth`) and the generative engine (`engine_checkpoint_*.pth`), both coordinated through `training_metadata.json`.
Dependency direction: helpers → `..datasets.*`, `..models.video_utils`, `..utils.*`; concrete loops additionally depend on `..engines.generative_models` (loaded lazily via `import_str`) and `..utils.recorder`.
