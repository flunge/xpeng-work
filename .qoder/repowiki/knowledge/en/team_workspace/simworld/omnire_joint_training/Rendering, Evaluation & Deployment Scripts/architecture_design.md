Two cooperating sub-packages with a clear dependency direction: `src/scripts/` (Python) depends on the sibling `reconic` package and its `sim_interface`, while `deploy_cmd/` (Bash) only invokes those Python scripts.

- `src/scripts/` is a flat collection of independent CLI tools, each bootstrapping imports by prepending parent directories to `sys.path` so they can reach `reconic.simulator.reconic_simulator.ReconicSimulator` and `scripts.render_sim`. The core rendering loop lives in `render_sim.py` (`render_origin`, `render_evaluate`, `render_lane_change`, `render_sine_waved_lane_change`) and is reused by `render_sim_feedforward.py`, which additionally sets `init_from_feedforward=True` on the simulator. `evaluate_model.py` defines an abstract `MetricBase` plus a concrete `PSNRMetric` (per-camera, per-timestamp JSON output with sky/object masks); `render_sim.py`'s `render_evaluate` orchestrates metric accumulation via `process_metric_calculation`. `utils/projection.py` and `xp_data_reader.py` are shared helpers; other files (`convert_large_model_to_ply.py`, `cvt_depth.py`, `data_valid.py`, `render_hil.py`, `render_harmonization.py`, `render_switch_car.py`) are one-shot utilities.
- `deploy_cmd/` contains thin Bash orchestration scripts: `train_cmd_feedforward.sh` mounts workspace paths, exports `TORCH_HOME`/`HF_HOME`, optionally rewrites configs for low-priority jobs, calls `render_sim_feedforward.py`, then uses `ffmpeg` filter_complex to stitch six camera videos into composite outputs. `eval_job.sh` wraps `eval_cmd.sh` inside a `fuyao deploy --site=fuyao_b1 --queue=adc-sim` call with a pinned Docker image, passing the checkpoint path down the chain. `render_harmonize_to_dds.sh` and `eval_cmd.sh` follow the same pattern of env setup → pip install editable → python invocation.

Boundary: this module has no internal package structure beyond `scripts/utils`; it is purely an execution surface over the `reconic` library.
