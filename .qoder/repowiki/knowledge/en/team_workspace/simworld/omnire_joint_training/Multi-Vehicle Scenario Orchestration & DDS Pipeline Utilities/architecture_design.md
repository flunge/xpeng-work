Flat collection of independent Python scripts under `reconic.multi_vehicle_utils`, each serving as a standalone CLI or importable helper rather than a cohesive package:
- `cloudsim_request.py` is the single HTTP client: it builds an HMAC-SHA256 signature (`X-Sign`) against per-domain secrets in `SECRETS` and posts JSON payloads to `*.xiaopeng.link` endpoints; all other modules call it via `from reconic.multi_vehicle_utils.cloudsim_request import cloudsim_request`.
- `query_scenario_event.py` consumes that client to implement two strategies — direct closeloop+openloop lookup when both IDs are present, or event-id-driven reverse lookup from openloop only — and normalizes results into a shared dict shape (`closeloop_scenario_id`, `event_id`, `threedgs_model_path`, `openloop_dds_result`, `vehicle_type`, `target_vehicle_full_name`).
- `create_multi_vehicle_scenario.py` is the top-level orchestrator: reads `e29_task_info_list.json`, checks OSS presence via `ossutil64 stat`, then publishes success/error messages to Kafka topic `cloudsim_3dgs[_dev]` through `ucp.tools.ucp_producer.UcpStreamingMsgProducer`.
- `dds_file_utils.py` handles OSS I/O (parsing `oss://bucket/key`, invoking `ossutil64 cp -r` for directories, optional lz4 decompression) and calls into `pipeline/ucp/download_file_from_oss2` by mutating `sys.path`; also compresses `recording*` files with `lz4` based on expected filenames.
- `encode_and_decode_utils.py` wraps `/usr/local/bin/camera_video_xp5_decoder` (env `CAMERA_VIDEO_DECODER`) in json/png/replace modes and composes side-by-side mp4s via ffmpeg's `hstack` filter.
- `png_to_h265.py` is a self-contained ffmpeg + `ThreadPoolExecutor` encoder that maps cam0–cam7 to named streams and uses NVIDIA NVENC (`hevc_nvenc`) with per-camera resolution/bitrate presets.
Dependency direction is one-way: higher-level scripts depend on lower-level helpers; no module imports another except `query_scenario_event -> cloudsim_request` and `create_multi_vehicle_scenario -> dds_file_utils` indirectly via task dicts. There is no `__init__.py`, so the directory is used purely as a namespace for these sibling scripts.
