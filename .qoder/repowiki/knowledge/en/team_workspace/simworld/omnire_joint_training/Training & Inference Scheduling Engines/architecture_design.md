Two scheduler implementations sit under `reconic/engines`:
- `serial_scheduler.SerialScheduler` (publicly exported via `__init__.py`) is the active implementation. It owns FIFO buffers (`training_pairs`, `inference_images`) and a `queue.Queue` (`novel_data`) of produced results. It does not own model logic; instead it delegates to an injected `generative_engine` through a thin protocol: `get_batch` / `training_forward`, `get_infer_batch` / `inference_forward`, plus `set_train`/`set_eval`, `save_checkpoint`, `resume_from_checkpoint`, and `training_batch_size`/`inference_batch_size` attributes. This makes the scheduler engine-agnostic.
- `generative_scheduler.py` is a Pyarmor 9.1.0 obfuscated build artifact of a parallel/generative variant; its source is not readable and it is not re-exported from `__init__.py`, so it is currently dead code in this package.
- The `pyarmor_runtime_000000` sub-package provides the runtime loader that the obfuscated file invokes at import time.
Dependency direction is one-way: schedulers depend on the generative-engine interface; the engine never imports back into engines. The module exposes only `SerialScheduler` as its public API.
