Two-process deployment driven by `pyproject.toml` entry points: a thin CLI (`tse.cli.main:app`, `typer`) installed as `tse`, and a long-lived daemon (`tse.server.agentd:main`, `uvicorn` + FastAPI) installed as `tse-agentd`. The daemon hosts two co-located async tasks — a FastAPI control API (`tse/server/control_api.py`) exposed over HTTP(S) and a Temporal worker (`tse.worker.run_worker`) that drives `ExperimentWorkflow` in `tse/workflows/experiment.py`. The workflow is a linear pipeline of Temporal activities under `tse/activities/`: build → submit → monitor → evaluate → report, with each step mirrored into a persistent status table via `tse/models/db.py` through the shared `mirror_status` activity. All external side effects (Git clone/build, cloud-sim API calls, simworld tool invocations, Feishu uploads) are isolated in activities; the workflow itself is pure orchestration using pydantic models from `tse/models/domain.py` for request/result contracts. Cross-cutting concerns live at package root: `config.py` (pydantic-settings), `constants.py` (Status enum / retry policies), `switches.py` (feature-flag registry consumed by both CLI `--set` parser and server), and `errors.py`. Optional dependency groups split install surface: `[client]` (typer+httpx), `[server]` (temporalio+fastapi+uvicorn+lark-oapi+requests), `[eval]` (pandas/matplotlib/numpy), `[ssh]` (fabric).
