Layered FastAPI application under `app/` with four internal layers:
- `interfaces/http`: FastAPI routers (`feishu_webhook.py`, `task_api.py`, `health.py`) that parse incoming Lark OpenAPI or compact events into `FeishuMessage` domain models and delegate to the orchestrator.
- `application`: `bootstrap.create_application()` wires an `ApplicationContainer` holding `TaskOrchestrator`, `SyncService`, `DailyAiHotScheduler`, `CommandParser`, and `ExecutorRegistry`; `orchestrator.TaskOrchestrator` owns a single async worker loop (`_worker_loop`) that pulls from an in-memory deque, runs each task in a thread via `asyncio.to_thread`, and drives status transitions through `domain.state_machine.TaskStateMachine`.
- `executors`: abstract `BaseExecutor` + `ExecutorRegistry` pattern; currently only `ChatReplyExecutor` is registered, but stubs exist for `preprocess`, `train`, `report_generate`, `logs_download` so new pipeline stages are added by implementing the interface and registering it.
- `infrastructure`: persistence via `SQLiteTaskRepository` (two DBs: `agent.db` for tasks/events, `tasks.db`), Feishu I/O via `RealFeishu*Client` classes backed by either the lark-cli auth adapter or a mock client depending on `settings.mode == 'cli'`, plus `LocalAgentClient` / `RepositoryAgentClient` for downstream service calls.
- `domain`: pure dataclasses/enums (`TaskRun`, `ExecutionResult`, `TaskStatus`, `TaskType`) plus the immutable transition table in `state_machine.py`.
Dependency direction is strictly inward: HTTP → application → executors → infrastructure; domain has no imports outside itself. The entrypoint `app/main.py` mounts routers, starts the worker loop, probes LLM health at startup, and exposes `/webhook/feishu/events`, `/tasks/*`, and `/health`.
