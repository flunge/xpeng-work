## O4 总体判断（90% · On track）

Q2 仿真算法组 O4 的核心能力从"链路打通验证"正式进入"可规模化生产、支撑发版准出"阶段。

**主要进展体现在四条线：**

1. **闭环链路可信化**——SIL 完成七车型泛化并产出 1000km+ 闭环结论（含广州 RC 路线 200km），HIL 台架扩展至 5 节点、500 job 95% 通过 CCES 检查，链路从"能跑"升级为"能复核"。
2. **渲染性能大幅提升**——NVFixer V3C/V3D 架构升级带来 PSNR +6~8dB，NVFixer ref 渲染性能比优化到 1:7.2、difix（MIG）1:14.8，HIL difix 单帧 123ms→40.7ms，3DGS 模型加载降低 72%，慢速/快速两套链路在台架跑通。
3. **规模化生产能力建立**——RC 长里程累计 2000km+（月底冲 2500km），极速模式全链路 100 分钟默认打开，UCP batch 上线支撑日产 70~100km，质检 gating 合入 IPS。
4. **算法预研初步验证**——场景编辑动态资产库 1700+、轨迹自动生成跑通 cut-in，AVM 鱼眼 cam9 适配，Inspatio-world LoRA 微调 pipeline 跑通。

**当前主要矛盾：**

Q2 算法组不再缺"能力"（链路都能跑），而是缺"可信度和准出标准"——
- 闭环 metric 连续数周不可用，导致场景集毕业和 RC 路线结论无法闭环（非我方根因，但阻塞我方交付）；
- SIL 与实车对齐率 46%，说明"能仿真"不等于"仿真结论可替代实车"；
- HIL 多次运行随机性尚未管理，复现率 50~74% 受模式和场景差异大；
- 极速模式有约 30% 复现率折损，仅适合紧急高优场景快速过一遍，不能替代正常模式。

**下阶段管理抓手（6 月集中纠偏）：**
- 锁死 6/20 闭环 metric 验收节点，用复现率 Agent + 人工抽检双轨先行，不等评估组；
- 建立 SIL/HIL/极速 三条链路各自的准出标准（复现率门槛、一致性门槛、适用场景边界）；
- 关键流程沉淀为文档 + skill + Agent，降低杜思聪离职后的单点依赖风险；
- 周二例会 + 双向清单堵住跨组同步缺口。

## KR1 仿真链路集成与等效性验证（80% · On track）

**目标**：实现 SIL 与 HIL 闭环链路常态化稳定运行，建立仿真-实车等效性验证体系。

---

### 项目 1：HIL 台架链路打通与稳定性验证

#### Plan

打通 XTest 标准框架接入 HIL 全链路（调度→渲染→CCES 评估→结论输出），从 3 节点扩展到 5 节点，验证系统级一致性。目标：运行成功率 > 95%，多次运行一致性 P95 横向误差 < 1m。

#### Do

- XTest 标准运行框架正式接入 HIL，广州 RC 路线闭环任务试运行；
- 完成 100 场景 × 5 次 = 500 case 一致性压测，**500 job 中 95% 通过 CCES 检查**；
- 77% case 的 P95 横向误差 < 1m，里程一致性中位数 0.21%；
- 节点从 3 扩展到 5（新增 004/005）；
- 排查并修复台架异常重启、PC 模块接收丢包、mflocalpose 跳帧、下游错帧等底层问题；
- 去掉 DDS 原始图像 topic 后，单场景耗时减少 11 秒；
- 将 YAML 位姿矩阵提取为 .npz，3DGS 模型加载耗时降低 72%（平均每 clip 约 5 秒）。

#### Check

**做得好的地方：**
1. 链路从"能跑"升级为"能压测出数字"，95% 通过率说明稳定性基本达标。
2. 底层问题逐个排查修复（丢包、跳帧、异常重启），说明工程化能力在建立。
3. 加载性能优化 72% 是实质收益，直接影响规模化效率。

**主要偏差：**
1. **一致性 77% 距准出标准（≥90%）仍有 13 个百分点差距**——根因是 14 个大偏差 case，初判 FM 起始速度/加速度 diff 导致轨迹分叉，不是渲染问题而是控制信号问题。
2. **HIL 底层调度存在多次运行随机性**——同一 case 不同次运行结果不完全一致，影响结论可重复性，当前缺乏随机性分类和容忍度标准。
3. **003 节点 shift 挂载 + 万兆网卡实验还在收尾**——下周三节点最后一轮稳定性压测结论仍待沉淀。

#### Act

**A1：14 个大偏差 case 专项根因归类**

| 根因类型 | 预期占比 | 解决方案 |
|---------|---------|---------|
| FM 起始速度/加速度 diff | ~50% | 与 FM 团队对齐初始状态注入方式 |
| Trigger time 偏移 | ~20% | 修正触发逻辑对齐精度 |
| 渲染帧错位 | ~15% | 时间戳对齐补偿 |
| 未知 | ~15% | 逐 case 复现、录制全链路 log |

**A2：建立 HIL 随机性管理规则**
- 同 case 跑 3 次取中位数作为基线结论；
- 3 次中方差超阈值的 case 标记为"不稳定"，不进入准出评估；
- 每周统计不稳定 case 比例，作为平台侧优化 KPI。

**A3：节点扩展验收标准**
- 5 节点全量压测 200 case × 3 次，一致性 ≥85% 视为合格；
- 6 月底输出带 PAT 的阶段性验收报告。

**Owner**：朱啸峰 / 瞿鑫宇 / 周蔚旭

### 项目 2：SIL 车型泛化与闭环验证

#### Plan

将 SIL 链路从单车型验证扩展到七款车型泛化（F30bes、h93aes、e29 等），跑通广州 RC 路线 1000km+ 闭环仿真，建立仿真-实车版本对齐结论体系。

#### Do

- 车型泛化从实验阶段进入规模化生产阶段，完成 F30bes、h93aes、e29 等多车型同场景闭环对比，结论与实车趋势一致；
- 完成斑马车衣与正常车衣互换闭环验证（验证外观差异不影响仿真决策一致性）；
- 广州 RC 路线完成 200km SIL 仿真，输出版本对齐结论；
- **对齐率 46%**，失真点主要集中在安全碰撞、不居中、行驶顿挫三类；
- 车型泛化功能代码已合入 Cloudsim，场景集管理流程工具 6/22 上线（入库/生产筛选/标签/车型泛化）。

#### Check

**做得好的地方：**
1. 七车型同场景对比且结论与实车趋势一致，说明 3DGS 泛化能力已经过初步验证。
2. 车衣互换实验是高价值的可控变量实验，证明渲染泛化的鲁棒性。
3. 场景集管理工具上线说明从手工作坊向工程化流程迈进。

**主要偏差：**
1. **对齐率仅 46%，距"可信替代实车"仍有明显距离**——意味着超过一半的仿真场景结论与实车不一致，当前无法作为 gating 依据。
2. **失真原因集中在三类（安全碰撞/不居中/顿挫），但尚未明确是渲染问题还是模型问题**——如果根因在模型侧，算法组难以独立解决。
3. **闭环 metric 不可用导致毕业标准无法闭环**——场景"能跑"但"不能毕业"，对外交付只能给"参考结论"而非"准出结论"。

#### Act

**A1：46% 对齐率专项拆解**

| 失真类型 | 占比 | 根因方向 | 解决方案 |
|---------|------|---------|---------|
| 安全碰撞 | ~30% | 动态物体行为不一致 | 交通流模型对齐+VRU表达优化 |
| 不居中 | ~35% | 横向控制偏差 | 排查FM横向参数差异 |
| 行驶顿挫 | ~25% | 纵向加速度不一致 | 初始状态注入精度+控制频率对齐 |
| 其他 | ~10% | 散点问题 | 逐case人工复盘 |

**A2：对齐率 KPI 分层**
- 第一层（可 gating）：对齐率 ≥ 80%，结论可直接用于版本准出；
- 第二层（可参考）：对齐率 50~80%，结论作为辅助参考，需实车复核；
- 第三层（不可用）：对齐率 < 50%，仅用于问题发现，不出对齐结论。

当前处于第三层边缘，6 月目标是推到第二层（≥50%）。

**A3：锁死 6/20 metric 验收**
- 如评估组 6/20 仍不交付闭环 metric，用复现率 Agent + 人工双轨做替代毕业标准先行推进；
- 每个毕业场景必须有 ≥3 次复现且人机一致率 ≥85% 才可发布。

**Owner**：杨星昊 / 裴健宏 / 云荟

## KR2 3DGS 渲染保真度与实时性能优化（95% · On track）

**目标**：攻坚 3DGS 渲染加速与模型优化，提升仿真复现率与实时响应能力。

---

### 项目 1：Difix（Non-Ref Diffusion）生产链路

#### Plan

Difix 作为默认快修模型，目标是在不需参考图的前提下修复 3DGS 渲染瑕疵，支撑大规模生产。核心指标：复现率稳定 ≥ 50%，极速模式中默认开启。

#### Do

- Difix 在极速模式中默认开启，随周四发版上线；
- 极速模式 108 case 复现 80 个，折损后约 50% 复现率；
- VLA 市场问题仿真闭环复现率 81.25%（26/32 起碰撞事故可复现）；
- 邓爽评估结论："虽略低于正常模式但可接受，后续默认打开"；
- 近点模糊是已知缺陷（受算力约束，GPU 暂无法分配给 Difix 额外训练）。

#### Check

**做得好的地方：**
1. Difix 默认开启 + VLA 复现率 81% 说明它已经从"实验"进入"生产工具"。
2. 复现率 50% 在极速模式（100 分钟全链路）约束下是可接受的——作为"快速过一遍"的第一道筛。

**主要偏差：**
1. **极速模式有约 30% 复现率折损**（正常模式 ~74%，极速 ~50%），核心原因是 Difix 近点模糊+feedforward 重建质量不如逐场景训练。
2. **近点模糊待改善但受算力约束无法短期解决**——这是一个需要排队等 GPU 的问题，不是算法问题。

#### Act

- 明确极速模式定位："漏斗第一关"——复现率不足的 case 自动进入正常模式重跑，不替代正常流程；
- GPU 到位后优先训 Difix 近点专项数据集（预估 2~3 天 64 卡训练）；
- 后续接入 FF Difix（feedforward + Difix 联合）提升极速下的近点表现。

**Owner**：周蔚旭 / 杨星昊

### 项目 2：NVFixer（Ref Diffusion）架构升级

#### Plan

NVFixer 带参考图修复，目标是高保真渲染——用真实参考图约束 Diffusion 输出，使修复后图像 PSNR 显著高于 Difix。架构从 V2 升级到 V3C/V3D，目标 PSNR 在 baseline 基础上 +6dB 以上。

#### Do

- **V3C 架构**：DIT block 中引入全局 self-attention，使修复区域与参考区域建立全局对应关系；
- **V3D 架构**：在 VAE decode 之前注入参考图 latent，让解码器直接看到参考信息；
- 实验结果：PSNR +6~8dB、LPIPS 明显改善，修复结果更趋近参考图；
- 周末提交 64 卡全量数据大批量训练，下周做变化仿真 FM 轨迹评测；
- Anyfix（带 ref 图版本）批量化 gating 测试集仿真评测已完成。

#### Check

**做得好的地方：**
1. PSNR +6~8dB 是非常大的提升（相当于视觉质量提高一个量级），说明 V3C/V3D 架构方向正确。
2. "全局 self-attention + latent 注入"双路线并行，增加了最终选型的确定性。

**主要偏差：**
1. **64 卡全量训练还在跑，结论待产出**——如果大规模数据下 V3C/V3D 的增益不如小实验，需要快速回退。
2. **带 ref 图方案的前提是"参考图可获取"**——对于新路线、新城市，参考图可能不存在或时效性差，需要明确适用边界。

#### Act

- 大批量训练结束后（预计本周末）立即做 FM 轨迹评测，对比 V3C vs V3D vs V2 baseline；
- 明确适用边界：ref 图获取成本 > X 分钟/km 时，退回 Difix；ref 图时效性 > 30 天需重采；
- 最终选型后（V3C 或 V3D）冻结架构，进入工程化集成。

**Owner**：周冯 / 杨星昊

### 项目 3：渲染性能压降

#### Plan

目标：3DGS + Ref Diffusion 单帧总渲染耗时性能比从 1:17 压降至 1:5 以内。双路径推进——算法侧（PTQ/算子融合/FMHA）+ 工程侧（H265 解码优化、DDS 链路精简）。

#### Do

- difix 渲染（**MIG 口径，apple-to-apple**）：baseline 1:17（510s/clip）→ EXP_5（ref 降分辨率）**1:14.8**、EXP_6 **1:15.5**；
- NVFixer ref 新版 TRT 轨迹评测 **1:7.2**（前序未优化 1:6.5），且 TRT 与 PyTorch 输出完全对齐；
- HIL difix 单帧从 v2 baseline 123ms 压到 v5（batch+noref+降分辨率+算子融合）40.7ms；
- 去掉 DDS 原始图像 topic，单场景耗时减少 11 秒；
- YAML→npz 加载优化，模型加载降低 72%；
- 实时模式不带 difix 已完成 10km 小数据跑通。

#### Check

**做得好的地方：**
1. NVFixer ref 从 1:6.5 优化到 1:7.2 且保证 TRT/PyTorch 数值对齐；HIL difix 单帧 123ms→40.7ms，工程优化扎实。
2. 多个优化手段叠加（PTQ/算子融合+加载优化+链路精简），说明团队在工程细节上投入足够。

**主要偏差：**
1. **距目标 1:5 仍有较大差距**（difix MIG 最好 1:14.8、NVFixer ref 1:7.2）——PTQ + FMHA 已到天花板，剩余空间不在算法侧而在系统侧。
2. **H265 解码耗时比目标（1:4.6→1:2）进展未明确**——解码侧是否有同等投入不清楚。

#### Act

- 解码侧专项：本周明确 H265→latent 转换的瓶颈环节，给出 1:2 的可行路径或明确声明不可达；
- 系统侧：与平台组协商增加 GPU stream 并行度，评估 1:5 是否可通过工程手段（而非算法）最后一公里达成；
- 如 1:5 短期不可达，建立分级耗时标准：快速模式 1:6 可接受、慢速模式 1:10 可接受。

**Owner**：瞿鑫宇 / 周冯

建立无参考图像质量评估（CLIP-IQA）自动质检 gating，在场景入库前自动筛除渲染质量差的 case，降低人工复核成本。目标：召回 ≥ 90%、精确 ≥ 80%。

- NQA 评测代码完成、测试中，已编包仿真；
- 与平台组对齐集成验证点与 JSON 输出格式，后续带包俊强跑完整链路评测；
- 质检 gating 已合入 IPS。

#### Check

**做得好的地方：**
1. 召回 90% 说明绝大多数差 case 能被捞出来。

- 精确度 < 80% 的场景类型做专项标注（50 case 人工复核），找出误杀模式；
- 与 Cloudsim 联调完成后，质检 gating 全量上线。

## KR3 规模化生产交付与算法预研能力建设（90% · On track）

**目标**：实现 3DGS 场景的规模化高质量生产与交付，前瞻布局下一代生成式仿真技术。

### 项目 1：RC 长里程生产

#### Plan

打通标准化量产通道，保障 8055km 长里程 RC 链路高质量交付。目标：月底累计 2500km，入库复现率 ≥ 50%，日产 70~100km。

#### Do

- RC 路线外业采集累计 2000km+（截至 6/17 新增约 1000km），月底目标 2500km；
- UCP batch 模式上线（5/29），日产目标 70~100km；
- 扶摇 SDK warmup 从 1.5h 压缩至 0.5h；
- 广州 RC + 新增路线持续生产，已完成 42 组中 36 组，产出 786 个 scenario；
- 长里程部分 case timeout（卡在 anti-distort 子进程未退出致父进程卡死），成功率 50~90% 不等；
- 76 条 subrun 因前期数据 pipeline 问题未入库（老数据格式不兼容），已在 oncall 群提单。

#### Check

**做得好的地方：**
1. UCP 日产 70~100km 说明量产通道已建立，不再是手工逐条跑。
2. warmup 从 1.5h→0.5h 直接影响每日启动成本。
3. 786 scenario 已产出，规模在持续增长。

**主要偏差：**
1. **距 8055km 全量交付仍有较大差距**——当前 2000km，按月底 2500km 计，Q2 结束时覆盖率约 31%，Q3 需要大幅加速。
2. **成功率 50~90% 波动过大**——anti-distort 卡死问题虽已定位但未彻底修复，影响生产效率。
3. **76 条历史数据未入库**——虽非新数据问题，但积压会影响覆盖率统计和完整性。

#### Act

**A1：anti-distort 卡死问题专项修复**
- 将多进程池改线程池（周蔚旭端午后测试结果待收）；
- 设置子进程 timeout + kill 机制，防止整体卡死；
- 失败 case 自动重试 + 超过 3 次失败标记为异常、人工复核。

**A2：月产能爬坡计划**

| 时间 | 目标 | 关键动作 |
|------|------|---------|
| 6 月底 | 2500km | 当前节奏维持 |
| 7 月底 | 5000km | UCP 并行数提升 + 新城市路线加入 |
| 8 月底 | 8000km+ | 4 城全覆盖，失败重跑自动化 |

**A3：76 条历史积压清理**

### 项目 2：极速模式（1 小时验证）

#### Plan

构建"1 小时极速验证"模式，将问题场景首轮验证基线交付时间压缩至 1 小时内。目标：全链路 ≤ 100 分钟，相对复现率 ≥ 50%，默认开启。

#### Do

- 极速模式 UCP 全链路约 **100 分钟**（含前处理 + feedforward 训练 + Difix 渲染 + 评估）；
- 相对复现率 74%（正常模式基线），极速绝对复现率约 50%；
- 默认打开，已随周四发版上线；
- 采用"漏斗策略"——极速先过一遍，复现率不足的 case 再进正常生产。

#### Check

**做得好的地方：**
1. 100 分钟全链路达成"1 小时级"验证目标（含前处理），从任务提交到结论产出 < 2h。
2. 漏斗策略设计合理——不强求极速完全替代正常流程，而是把它定位为快速筛。
3. 默认开启 + 发版上线说明已经是生产工具而非实验品。

**主要偏差：**
1. **30% 复现率折损是结构性问题**——根因在 feedforward 重建质量（跳过逐场景训练），不是 Difix 参数问题。
2. **仅适用紧急高优场景**——对于 gating 数据集，极速模式结论不可信。

#### Act

- 明确文档标注：极速模式的结论标记为"参考级"，不作为准出依据；
- 后续接入 NVFixer 最新版本（V3C/V3D），评估是否可在极速链路中用轻量 ref 图补偿折损；
- 统计极速→正常模式的"晋级率"（极速过的 case 在正常模式也过的比例），作为极速可信度的持续 KPI。

**Owner**：周蔚旭 / 杨星昊

## 跨 KR 共性问题

### 从"能跑通"到"可准出"还差一步

Q2 各项目都已跑通（SIL/HIL/极速/慢速/质检/场景编辑），但跑通不等于可信。下阶段不能只看"是否跑通"，而要看：

- 结论是否稳定（多次运行一致）？
- 结论是否可信（与实车对齐）？
- 结论是否可重复（不同人跑同一 case 得同一结论）？
- 结论是否能 gating（有明确的 pass/fail 标准）？

### 评估组 metric 依赖是系统性风险

闭环 metric 不可用不是偶发事件——它导致我方已生产的场景无法毕业、RC 路线结论无法对外输出、对齐率无法量化改善。这是跨组依赖问题，需要在部门层面管理。

**应对**：不等不靠，用复现率 Agent + 人工抽检双轨建立替代准出机制先行推进。
