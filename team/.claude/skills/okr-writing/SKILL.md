# OKR 撰写规则

## 🔴 第一铁律：不准抄上季度的 OKR 做"简单延展"

**新季度的 OKR 必须基于「上季度末实际在做的项目」+「规划中想做的事」重新推导，而不是把上季度 OKR 改几个数字。**

- 写之前**必读**：上季度末的项目 ledger（`projects/<track>/*/ledger.md`）、最近 2-4 周的会议纪要/日报、双周会上老板的方向性输入
- 判断标准：如果新 OKR 和旧 OKR 长得像同一篇、只是数字往上调了 → 不合格，推倒重来
- 反例（2026-06-30 实际发生）：Q3 OKR 照搬 Q2 的 4 个 O，把"漏报率 20%"等指标简单延续，没有把 Q2 末新做的项目（车型泛化三参数扫描、AB Review Agent、NVFixer V3C/V3D、CLIP-IQA gating 等）纳入

## O（Objective）的写法 — 对齐 Q2 实际表述（范本 wiki `I9wCwVRFqiteBEkh3h1cXtibnDg`）

格式：**`【两到四字标题词】+ 强动词领衔的战略长句`**，书面、有气势，不是口语化的"做成 XX"。

- 用强动词开头：攻坚 / 实现 / 规模化 / 突破 / 驱动 / 建立 / 绑定…
- 点明这条线本季度的战略终局 + 战略价值，一句话讲完
- 例（Q2 原文）：「**【渲染优化】攻坚 3DGS 渲染加速与模型优化，提升仿真复现率与实时响应能力。**」
- 例（Q2 原文）：「**【生产交付】规模化 3DGS 场景生产与高质量交付，支撑长里程与高价值 Corner Case 仿真验证**」
- 反例（太口语，不要）：「把仿真场景做成'按天可交付'的产线」「让 SIL 成为标准、渲染又快又真」

## KR（Key Result）的写法 — 对齐 Q2 实际表述

格式：**`KRn：动词短语小标题（加粗）。 完整描述句——怎么做 + 业务目标 + 加粗的量化数字`**

三要素缺一不可：
1. **这是一件什么事**：说清做的内容（不是抽象口号）
2. **要达成什么业务目标**：为什么做、对业务/发版意味着什么
3. **数字量化**：🔴 **没有数字的 OKR 就是耍流氓**。百分比/效率比/case 数/里程/覆盖数/准确率/时间节点，**关键数字加粗**

- 每个 O 下 3-4 条；数字可溯源 ledger 基线 + 合理爬坡，不拍脑门
- 例（Q2 原文）：「**KR3：突破 SIL 底层性能瓶颈，强力支撑 1:25 全局提效目标。** 聚焦模型渲染与视频解码双链路的深度压榨，将 3DGS + Ref Diffusion 单帧总渲染耗时压降至 **0.5 秒内**（性能比 1:13 跃升至 **1:5**），并实现 H265 解码耗时比由 1:4.6 提效至 **1:2**。」
- 写法骨架：`KRn：<强动词>+<小标题>。 <怎么做的一句话>，<量化指标 X→Y / 达到 Z>，<对业务/发版的意义>。`

## 🔴🔴 第零铁律：所有数字必须溯源（2026-06-30，无例外）

**OKR/周报/双周报/任何汇报里出现的每一个数字，写之前必须回 ledger/源文档逐个核对出处，不能凭记忆套用。**

- "X→Y" 的**基线 X 和目标 Y 都要单独溯源**——不是只验目标合理性。基线写错=整条 KR 失信。
- 反例（2026-06-30 实犯）：把极速模式折算值 51.8%（=普通基线70%×极速相对74%）当成"常规模式复现率基线"，写"常规模式 51.8%→65%"。常规模式真实基线是 70%。一个数字张冠李戴，被老板当场抓出。
- 同一指标在不同场景/模式下有不同数值（如复现率：常规 70% vs 极速 51.8%），**必须确认你引的是哪个场景的数**，别串用。
- 拿不准出处的数字：要么找到 ledger 原文，要么不写，绝不编造或"差不多套一个"。

## 🔴 KR 优先级：按重要性排序，KR1 最高（2026-06-30）

- 每个 O 下的 KR **按优先级从高到低排**，KR1 是这条线本季度最重要、最该先打的仗。
- 写完后自问："如果这季度只能完成一条 KR，是不是 KR1？" 不是就重排。
- 例（O2 SIL）：车型泛化"用仿真替代多车型实车采集、支撑 630 多车型发版"是 SIL 第一优先级 → 必须排 KR1，而非渲染提速/质检之后。

## 🔴 七条红线（2026-06-30 李坤逐条纠正后沉淀，违反任一条即不合格）

1. **数字用「当前真实基线」，不抄上季度 OKR 的目标值。** 写新季度 KR 前，先查 ledger / 日报拿到「今天的实际值」（已累积多少、产能多少、复现率多少），基于真实基线设合理爬坡。反例：写"里程从 1wkm 扩到 2wkm"——1wkm 是 Q2 OKR 的目标值不是实际值（实际只产了约 2000km），属于"不切实际地照抄"。

2. **不把上下游的工作写成自己的 KR。** 别的组（大数据、平台、引擎、台架）做的事，我们能写的只有「监控 + 推动」。反例：写"攻坚大数据 35% 折损"——大数据不是本组。正确：写"建 XX 全链路看板监控折损、推动上下游把留存率提到 XX%"。

3. **不写自己无法控制的指标。** KR 的量化指标必须是本组能直接影响/交付的。反例：写"日均提交 ≥50 job"——别人提不提交我们控制不了。正确：写本组能交付的能力（覆盖范围、复现率、交互简化程度）。

4. **数量不是目标，能力才是目标。** 对生产/泛化类业务，写"可用率、自动化能力、泛化能力"，不要写"产 N 个 case"。反例："月产 ≥100 case"。正确："泛化可用率达 XX%、自动化生成能力覆盖 XX 类场景无需人工干预"。要对业务模块有深入理解再下指标。

5. **KR 里不写实验细节/中间过程。** 架构代号、消融数据（如 +8dB / V3D VAE decoder 后注入）属于实验记录，不进 OKR。OKR 只写「要达成的结果指标」。

6. **写项目的「目的和目标」，不写「实现手段」。** 具体技术方案（如 CLIP-IQA、某个模型）只是当前手段，Q3 可能换方案。KR 写"要解决什么问题、达成什么目标"，手段最多作为补充。反例："CLIP-IQA 接入评测链路"。正确："建立图像质量自动化卡口，瑕疵召回 99%/准确率 90%，不合格图像不进评测"。

7. **该有的核心业务指标不能漏。** 如果一条线有标志性指标（如仿真的「复现率」），必须显式设目标，且区分场景（常规模式生产 vs 极速模式生产的复现率分别设目标）。

8. **KR 要短，一句话讲清结果+数字，不写"为什么有意义"。** KR 是结果承诺，不是战略论证。**禁止**把"直面 XX 矛盾""用 X 换 Y""把 A 变成 B"这类解释性长句塞进 KR——战略意义放 ledger/汇报叙事里。一条 KR 控制在 ~50-80 字：动词小标题 + 量化结果即可。反例（啰嗦 180 字）："以车型泛化用仿真替代多车型实车采集……直面'有限资源 vs 几何增长'核心矛盾——以 3DGS 外参迁移……用算力换成本……把实车盲测变为同速同场景可控对照。" 正例（102 字）："车型泛化用仿真替代多车型实车采集，支撑 630 多车型发版。Q3 覆盖 ≥30 款车型、打通 daily 评测闭环，输出车型可共用 3DGS 参数的聚类规律。"

## 仿真算法组的 4 个 O（2026-06-30 李坤确认的固定框架）

按此顺序，覆盖全部在做内容，不要遗漏：

| O | 方向 | 范围界定 |
|---|---|---|
| O1 | **场景产线** | 3DGS 场景生产、长里程交付、极速模式、场景编辑/CornerCase、质检 gating |
| O2 | **SIL** | 车型泛化等等——**所有在 SIL 上验证的内容都算这里**（渲染算法 Fixer/NVFixer、CLIP-IQA、车型泛化、RC 路线 SIL 验证、红绿灯等业务验证） |
| O3 | **HIL** | **Q3 重要目标 = 绑定发版 gating 的业务**；链路稳定性、效率比、复现率、台架规模化、慢速模式 |
| O4 | **预研 & Agents** | Diffusion/Feedforward/WorldModel/场景泛化预研 + 复现率/Diff/Prompt/环境构建等 AI Agent |

## 写作流程

1. 读上季度末 ledger + 近期会议 → 列出「在做的」+「规划想做的」项目清单，按 4 个 O 归类
2. 每个 O 提炼一句振奋人心的 Objective
3. 每个 O 下选 3-4 件最关键的事写成 KR，每条配可溯源的量化指标
4. 自检（见下）后写入文档

## QA 自检清单

```
□ 没有照抄上季度 OKR，每条 KR 都对应 Q2 末真实在做/规划的项目
□ 4 个 O 顺序：场景产线 / SIL / HIL / 预研&Agents
□ 每个 O 是振奋人心的大目标，不是任务描述
□ 每条 KR 三要素齐全：做什么 + 业务目标 + 数字量化
□ 🔴 每条 KR 都有数字（无数字=不合格）
□ 数字可溯源到 ledger 基线或合理爬坡，不编造
□ HIL 的 O 体现「绑定发版 gating 业务」
□ SIL 的 O 涵盖所有在 SIL 上验证的内容（含车型泛化）
```
