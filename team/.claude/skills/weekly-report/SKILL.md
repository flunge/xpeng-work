## 📌 从李坤本人修改中学到的写法（2026-07-01 对比双周报 A42Gd vs 他改后的 wiki 3DGS 节，沉淀）

李坤把我的对外双周报搬进大会 wiki 时做了实质优化，不是搬运。这些是他偏好的**内部技术材料**写法，写这类材料时默认套用：

1. **"目标→现状→进展→计划"四段，比双周报的"目标→进展→计划"多一段「现状」**：现状 = 本项目从哪个基线出发（如"CloudSim 已上线支持开环+闭环""单点需求不断 G02|D03 不加速""复现率校验还没自动化"）。少了现状，读者不知道进展是站在什么起点上。**内部材料写现状，对外双周报可省。**

2. **数字要给「提升了多少（delta）」+「下一步具体技术路线」，不能只报一个结果值**：反例（我写的）"效率比 1:7.2、TRT 与 PyTorch 已对齐"——只有结果、偏空。正解（他改的）"渲染效果相比 Difix **增加 1.2dB PSNR**、效率比 1:7.2（Difix 1:17）；**下一步 PTQ/QAT + 轻量化 VAE（tinyVAE/LiteVAE）**"。即：① 相对基线提升多少 ② 接下来打算用什么具体技术。我的老毛病是只报终值、不给 delta 和路线。

3. **能画成「原方案 vs 新方案」对比图的机制，就别用文字描述**：慢速模式我用一句话"ref 图预编码 latent、链路直读"；他拆成`原方案[图：生产-渲染串行、每次下载 reference 解码 VAE]→新方案[图：生产同时产出 reference latent、渲染时加载]`两图 + 极简文字。机制/改造类内容优先图示对比。

4. **措辞更贴业务、更准，且补我漏的进展**：如"发版准出把关"→"可用作后续版本的**绝对 gating 指标**"；"找规律"→"**车型|车衣|外参多维度**找规律"（点明维度）；HIL 补"已导入 **6 类闭环 metric、结果符合预期**"、RC 补"可保证 **subrun 15 分钟无中断**"（质量佐证）。写之前把项目最新进展的量化佐证捞全，别只写我记得的那几条。

5. **文体区分：对外双周报 vs 内部大会材料不是同一套**。双周报有 🔴🟡 独立风险块、要极简；内部材料把关键风险并进「现状」、可带更多技术细节（PSNR/PTQ/tinyVAE 这类内部术语在内部材料里正常，对外双周报才需掂量）。**搬运 ≠ 复制**：把双周报放进内部材料时，按上面 1-4 补现状/delta/路线/对比图，而不是原样贴。

## 工作流程

1. **Collect** — 按优先级顺序串行采集（不可跳过任何一步）：
   - **Step 1【必须】Q2 Wiki**：遍历 `https://xiaopeng.feishu.cn/wiki/SBUYwm8Lri9aJ6kmexFcBAuGnlh` 中本周时间窗口（上周三→本周二）的进展列——Wiki 是表格格式，每一天是一列（如 0609 列），逐列读完；再读所有嵌套链接文档全文
   - **Step 2【必须】会议纪要+逐字稿**：遍历本周时间窗口内所有核心日会、组内日会、其他会议的智能纪要 **+** 文字记录（逐字稿），两者都要读，缺一不可
   - **Step 3【必须】聊天记录**：遍历本周时间窗口内的群聊消息和 p2p 消息
   - **⚠️ Memory 文件仅供背景理解，不是数据源**：memory/people/*.md、memory/projects/*.md 用于理解人和项目背景，不能替代上面三步直接编写周报内容
2. **Analyze** — 逐轨提取：场景&生产 / SIL / HIL / Agents 的结构化进展
3. **Synthesize** — 合成报告：按高炳涛偏好格式
4. **Output** — 写入飞书文档 + 本地存档

## 🔴 飞书智能纪要 ASR 术语清洗（2026.06.29 沉淀）

**问题**：飞书智能纪要由 AI 从会议录音生成，对英文缩写/项目代号识别经常出错。**直接照搬原文写进周报 = 把识别错误传播给管理层**。

**写周报前必须做术语映射清洗**。已知 ASR 错词对照表：

| 智能纪要原文（错） | 实际是 | 证据 |
|---|---|---|
| 黑友 / 黑耀 / Hill / HEEL / HELL / HERA | **HIL** | "HIL"读"嗨欧"被音译。6/29 Demo 写"黑友链路 5 节点机房" |
| Seal / SEAL | **SIL** | "SIL"读"塞欧"被听成"Seal"。6/22 例会"Seal/HIL 全局开关" |
| 黑油 / 黑友站台 | **HIL 台架** | 同上 |
| AA Docker | **AI Docker** 或 **agent Docker** | 6/24 例会原文 |
| Hill 链路 / Hail 链路 | **HIL 链路** | 同上 |
| AP8 | **FP8** | 6/25 WM 同步上下文（推理精度） |
| Ai 一笑 / 曾一笑 | **曾一笑**（人名） | 多次确认 |
| 化隆 / 华龙 | 需上下文判断（可能是项目代号或人名）— **必须问** | — |

**规则**：
1. 写周报前对照表把"原文术语"换成"标准术语"
2. 表里没有但是看起来像 ASR 错的（出现频率高 + 上下文明显是某个英文缩写） — **不要猜，问用户确认后加入表**
3. 报告里**不准同时出现两种叫法**（如同一份周报里"HIL 链路"和"黑友链路"并存 = bug）
4. 发现新的 ASR 错词 → 用户确认后增补到此表

**反例（2026-06-29 实际发生）**：周报 HIL 行同时出现"HIL 链路"和"黑友链路"，Fixer 优化提到"Seal 链路"——管理层看见会以为是新项目，实际全是 HIL/SIL。用户直接问"黑友/Seal 是什么"才暴露。

## 🔴 读文档的颗粒度铁律（2026.06.29 沉淀）

**规则**：**相关的内容必须全部读完**，不挑核心/附属、不挑最近/早一周、不挑主线/分支。只有**明确判定无关**的会议（HR 干部宣导、新人入职欢迎、跨业务的领导汇报）才能跳过，跳过必须在主对话区写明"跳过 X 场，理由 Y"。

**默认动作**：
1. 从 daily-sync 的 `meeting_docs` 列**全部**会议清单（按日期 + 主题 + token）贴在主对话区
2. **🔴 daily-sync 缺当天/前一天必须用 `lark-cli vc +search` 兜底**（daily-sync 是 22:07 跑的批处理，前一天的 .json 通常缺当天会议；今天的 .json 完全没有）。兜底命令：
   ```
   lark-cli vc +search --query "<关键词>" --start <D-1> --end <D+1> --page-size 30
   ```
   关键词至少跑 4 次：仿真 / HIL / Agent / Demo（覆盖典型场景）。找到的 meeting_id 用 `vc +detail` 拿 minute_token，再用 `drive +search --query "<会议标题>"` 找到智能纪要 + 逐字稿的 docx token。
3. 标记每场会议「读 / 跳过」状态
4. **每场会议必读智能纪要 + 逐字稿两份**（智能纪要是摘要，逐字稿是原话；缺逐字稿 = 没读）
5. 时间窗口横跨多周时，**所有 W 段周表都要读**（如 6/24-6/30 跨 W26+W27，两个 block 都得读）

**反模式**（已发生过的）：
- 跳过"每日例会"觉得是附属 → 漏掉技术细节最详的源
- 跳过"专项会议"（HIL 节点会/WM 同步/算法 idea）觉得不归 4 大轨 → 漏掉跨链路决策
- 只读 W27 不读 W26，因为周报标题是"本周" → 漏掉本周一半的天数

## 🔴 历史交互：读完后必须做的"4 找"（2026.06.29 沉淀）

读源文档不是消化吸收，是**信息建库 + 主动找问题**。每读完一份源，对照已读源 + 历史记忆做 4 件事：

### 找①：承诺追踪
- 老板（高炳涛 / 邓爽 / 徐林鵾）每提一个**时间点的待办**（"下周一前提交"/"7/2 上线"/"明天确认"），记入「承诺追踪表」
- 写周报时**逐条检查**：本周该到期的承诺，源文档里有没有兑现/延期/转向？没出现的要追问

### 找②：决策对比（矛盾检测）
- 同一议题在不同时间点的两次发言是否矛盾？
- 例：6/22 高炳涛说"代码 Q3 大调整、各组下周一前提整改计划"，6/29 改口"每周扫问题直接改"——**这种 180° 转向必须在周报里显式呈现**，不能只写最新版盖掉旧版

### 找③：静默检测
- 上次周报里出现过的项目/承诺/风险，**本周完全没出现** = 危险信号（要么完成了没说，要么停滞了没说）
- 写前对照上周周报清单，每个静默项追问一次

### 找④：缺口检测
- 本周某事提到"明天验证"/"周五给报告"/"会后对齐"——**对应日期之后的源文档读了没？**
- 缺一个就意味着这条线索是断的，要么补读要么标"待跟进"

**反模式**（已发生过的）：
- 6/22 老板说"AI agent 明确落地计划节点+量化收益" → 周报 Agents 行只写各 agent 准确率，完全没体现"量化收益"维度
- 6/17 茶水间提"早启动机制、周固定例会" → 后续两周完全没追踪是否落地
- 6/24 决策"VAE 改 TinyVAE" → 6/25 后续有没有结论？没有追

## 🔴🔴 逐条回源自查：不是自问，是动手查（2026-06-30 血泪沉淀，最高优先级）

> **根因反思**：过去反复被用户一条条抓出"旧的当新的/脑补/时态错误"，根子是——我写周报靠"记忆+旧 ledger 拼一版"，自检当走过场（心里默念 5 问就算过），**从不真的把每一条回本周源文档搜一遍**。等用户指某条我才去查那条 = 用用户当校验器。这是不可接受的。

**写完后（发给用户前），必须机械执行以下动作，不能跳、不能"感觉没问题"就过**：

1. **把报告拆成逐条 bullet**（每个项目的 进展/现状/风险/计划 各算一条），编号列出。
2. **把本周期全部源头拉成语料**：两份日报（`Wu6yw...` 核心日会日报 + `SBUYw...` 作战表本周 W## 列）+ 本周每场会议纪要/逐字稿 + chat-log 本周段，`grep` 进一个工作区临时文件。
3. **逐条 grep 回源**：每条 bullet 的关键事实（数字、状态、动作、时间）拿去源语料里搜。对每条明确判定：
   - ✅ 本周源文搜得到、且表述一致 → 保留
   - ⚠️ 搜得到但我改了述（数字/修饰词/确定性）→ 改回源文原述（反例："前端界面"是脑补加的、"下周上线"源文是"正在调研"、"本周试用"源文是"明日试用"）
   - ❌ 本周源文搜不到 → 是旧 ledger/记忆/非正式来源，**删**
4. **把逐条核对结论列给用户看**（哪条留、哪条改、哪条删，各自源在哪），而不是只说"校验通过"。

**这一步没做 = 没写完。** check_report.py 只能抓固定模式（黑话/溯源/重复/排除项），**抓不出"这句是不是本周的"——那必须靠这个逐条回源动作**。

## 🔴 自检触发条件（2026.06.29 沉淀，写完必做）

写完后**必须自问 5 个问题**，任一答不上 = 重写：

```
□ Q1 老板本周在核心日会上具体说了哪几句话？我的报告里能对得上吗？
□ Q2 上周周报里列的承诺/风险，本周报告里有兑现进展吗？没出现的能解释吗？
□ Q3 本周窗口内的所有会议（不止核心日会），我清单列全了吗？读全了吗？
□ Q4 本周报告里有哪些项是"我以为有进展所以写了进去"，但实际源文档里没明确写？
□ Q5 报告里每个具体数字（百分比/case 数/耗时比）能指到具体哪天哪场会哪句原话？
```

**反模式**（已发生过的）：
- 写完不自检，等用户问"6/24 每日例会呢？"才发现漏
- 数字凭"印象差不多"写，被追问溯源时答不上
- 把源数据"1300+ scenario"写成"近 1500 条数据"——凑数改述 = 脑补
- 把旧实验数据（6/13 一致性 100×5）改写换述后放进本周进展——旧数据冒充新进展
- "1000 工作台"写进风险但不解释含义——用户看不懂 = 没写好
- 🔴 把内部溯源标注 `（来源：6/25 节点会 R3BzdeiFo...）` 写进给老板看的正文——**溯源是写之前自己要做到的功课，不是糊进报告给老板看的**。老板要的是干净结论，不是 token 和会议指代
- 🔴 内部黑话/词salad直接抛给老板：塔包、expand 框架、DSOP、RTM topic、"三参数敏感性"、"老框架跑起来"——连懂行的也得猜

**数字/术语/溯源三铁律（2026-06-30 追加，已落 check_report.py 机器拦截）**：
1. **数字不改述**：源文档说 1300+，报告只能写 1300+，不能写"近 1500"、"约 1500"；旧周实验结论若本周无新数据，只能进"现状"标"此前已验证"，禁止冒充"进展"。
2. **黑话说人话**：高炳涛是仿真部 P9，scenario/metric/latent/batch/PAT 他懂、可直接用；但"连懂行的也得猜"的内部缩写/口语指代（塔包→车型标定压缩包、expand 框架→旧评测框架、DSOP/RTM→上游数据格式、工作台→仿真任务）必须翻成人话或首次出现加括号解释。判据：这个词换个仿真部别的组的人看，懂不懂？不懂就改。
3. **溯源是功课不是正文**：写每条之前我自己必须能指到源（哪天哪场会哪句话），但**正文里绝不出现** `来源：`、token、`6/25 节点会` 这类指代。报告只给老板看干净的「进展/现状/风险/计划」。需要追溯时我自己心里/记忆库里有，不写进文档。

## 🔴🔴 三层工作法：收集 → 过滤相关性 → 结构化（2026-06-30 沉淀，必须分层，禁止一股脑丢）

写周报/双周报**必须分三步走，不能跳过第 2 步直接堆内容**（这是 6/30 写出一堆"垃圾"的根因）：

### 第 1 层 收集：按项目颗粒度归集
把窗口内相关会议/文档/聊天的进展，**先归到各项目 ledger 颗粒度**（极速模式、RC 长里程、车型泛化…），不要边读边往报告里抄。

### 🔑 飞书 XML 格式写表格（2026.06.11 关键教训）

**问题**：`--doc-format markdown` 写 HTML `<table>` → 飞书 markdown 解析器吃掉表格标签 → 四轨内容变成扁平文本，月目标列丢失。

**正确做法**：用飞书原生 XML 格式（`lark-cli` 默认 `--doc-format xml`）：

```xml
<table><colgroup><col width="100"/><col width="300"/><col width="600"/></colgroup><tbody>
<tr><td vertical-align="middle"><p align="center"><b>链路</b></p></td>
    <td vertical-align="middle"><p align="center"><b>月目标</b></p></td>
    <td vertical-align="middle"><p align="center"><b>核心进展</b></p></td></tr>
<tr><td vertical-align="middle"><p align="center"><b>场景&amp;生产</b></p></td>
    <td><ul><li>月目标1</li><li>月目标2</li></ul></td>
    <td><ul><li>进展1</li><li>进展2</li></ul></td></tr>
</tbody></table>
```

- **列宽固定**：100 / 300 / 600 px（2026.06.29 沉淀，不要再用 180/420）
- **首行（表头）+ 首列（链路名）**：粗体 + 居中（`<p align="center"><b>...</b></p>`），其他单元格保持左对齐
- 🔴 **任何表格都必须带 `<colgroup>` 列宽——重建/替换表格最容易漏（2026-07-01 连漏 2 次被用户抓）**。列宽按列数配：3 列周报=100/300/600；**Q3 作战表 5 列（周标签+4 优先级）= `100/500/500/500/500`（照 W27 原表规格，不要自己拍数值）**。改表前先 `--detail full` 看已有同类表的列宽、照抄，别自创（我曾自拍 90/235 导致与 W27 的 100/500 不齐被用户抓）。
- 🔴 **普通 `fetch` 不回显 col width（colgroup 显示成空 `<col/>`）——查列宽必须用 `--detail full`**。别用普通 fetch 的空 col 判断"列宽没生效"（那是显示省略，值其实在）；`--detail full` 才能拿到真实 `width="500"`。

- `--command overwrite --content "<xml内容>"` 不指定 `--doc-format`（默认就是 xml）
- `<td>` 内可以嵌套 `<ul><li>`、`<b>`、`<img>`、`<cite>`
- `&amp;` 转义 `&`，`&lt;` 转义 `<`

### 🔑 用 `<cite>` 标签 @人（2026.06.11 关键教训）

**问题**：写"杨星昊"纯文本 → 高炳涛看到的只是名字，不能点击跳转。

**正确做法**：用飞书 `<cite>` 标签：

```xml
<cite type="user" user-id="ou_b41c33085d2e629fbdff0c555cae0a3f" user-name="杨星昊"></cite>
```

- `user-id` 从哪儿来：会议纪要 doc 的参会人列表（`<cite type="user" ...>`）、memory 文件 `team-members.md`
- 全组13人的 user-id 已知，可直接复用
- 飞书渲染后显示为可点击的 @人名
- **注意**：`<cite>` 是自闭合标签，后面紧跟冒号或空格时需要额外处理，否则可能粘连。建议格式：`<cite ...></cite>：`

### 怎么"增加高炳涛感兴趣的内容"

不另开章节，而是在核心进展列里自然融入：
- AI 使用：在相关条目后加一句"（AI辅助xxx）"或标注模型/成本
- 代码质量：有重构/清理/审计时显式写出
- 时间节点：每条进展都带上具体日期，不写"近期""下周"
- 闭环：每条进展说清楚"做了什么→结论是什么→下一步是什么"

### 图片处理

用 image-agent 的搬图流程：
1. 从会议纪要 doc 下载图片（`docs +media-download --token <src>`）
2. 上传到个人文档（`docs +media-upload --file ./img --doc-id <id> --parent-node <id> --parent-type docx_image`）
3. 用 `block_insert_after` 在占位文本后插入 `<img src="file_token" mime="..."/>`
4. **插入后删除占位文本 block**（否则图文重复，显脏）

注意：API 直接插入的图可能显示为方形框，已知限制。

### 表格列格式（从 W7 起统一格式）
```
链路 | 目标 | 月目标 | 核心进展-MMDD | 核心进展-MMDD | ...
```
- 每天一列，新列在右侧追加，格式为 `核心进展-MMDD`（如 `核心进展-0615`）
- **直接读全文会因截断丢失最新列** — 必须用 `--scope keyword` 定向搜索

### ⚠️ 已知陷阱（2026-06-15 测试验证）

| 陷阱 | 说明 |
|------|------|
| `--scope keyword` 只返回列标题行 | keyword 命中的是表头单元格，`<excerpt>` 只包含该行，不包含数据行（四轨名称和进展都在其他行）。**不能用 keyword 搜索来读取进展内容** |
| 报告窗口横跨两个 W 节 | 例如 Jun 11(Thu)-17(Wed) 中，0611 在 W10，0612-0617 在 W11。需同时读覆盖窗口的所有 W 节 |
| `--scope section` 返回整个 synced-source | 请求 W11 的 section 会返回 W6-W13 全部内容（~200KB）。需客户端按 H4 block_id 过滤到目标周次 |
| 部分日期无 wiki 条目 | 周末通常不更新，节假日亦然。窗口内缺日期属正常现象 |

### 正确读取方法（已测试可行，2026-06-15）

```python
from datetime import date, timedelta
import subprocess, json, re

def get_report_window(today):
    """计算汇报窗口：上周三 → 本周二（7天）。在任何星期几运行均正确。"""
    days_since_wed = (today.weekday() - 2) % 7   # Wed=0, Thu=1, ..., Tue=6
    most_recent_wed = today - timedelta(days=days_since_wed)
    last_wed = most_recent_wed - timedelta(days=7)
    this_tue = most_recent_wed - timedelta(days=1)
    d, dates = last_wed, []
    while d <= this_tue:
        dates.append(d.strftime('%m%d'))
        d += timedelta(days=1)
    return last_wed, this_tue, dates

def extract_week_section(full_content, h4_block_id):
    """从 section 全文中按 H4 block_id 提取对应周次内容。"""
    start_pat = re.compile(r'<h4 id="' + h4_block_id + r'">')
    m = start_pat.search(full_content)
    if not m:
        return ''
    start = m.start()
    rest = full_content[start + 10:]
    m2 = re.search(r'<h4 id="[^"]+">', rest)
    return full_content[start: start + 10 + m2.start()] if m2 else full_content[start:]

# Step 1: 获取整个"周目标&进展"章节（section_block_id 固定）
result = subprocess.run(
    ['lark-cli', 'docs', '+fetch', '--api-version', 'v2',
     '--doc', 'https://xiaopeng.feishu.cn/wiki/SBUYwm8Lri9aJ6kmexFcBAuGnlh',
     '--scope', 'section', '--start-block-id', 'STymdPgAJorwXvxjsrWcJKTMnDe',
     '--format', 'json'],
    capture_output=True, text=True
)
content = json.loads(result.stdout)['data']['document']['content']

# Step 2: 找所有 H4 blocks
h4s = re.findall(r'<h4 id="([^"]+)">([^<]+)</h4>', content)

# Step 3: 计算目标日期，找覆盖的 W 节
_, _, target_dates = get_report_window(date.today())
combined_content = ''
matched_weeks = []
for bid, label in h4s:
    week_content = extract_week_section(content, bid)
    cols = set(re.findall(r'核心进展-(\d{4})', week_content))
    if cols & set(target_dates):
        combined_content += week_content
        matched_weeks.append(label.strip())

# Step 4: 提取嵌套文档 IDs（去重）
nested_doc_ids = list(dict.fromkeys(re.findall(r'doc-id="([^"]+)"', combined_content)))

# Step 5: 逐一 fetch 嵌套文档
for doc_id in nested_doc_ids:
    subprocess.run(
        ['lark-cli', 'docs', '+fetch', '--api-version', 'v2',
         '--doc', doc_id, '--doc-format', 'markdown', '--format', 'json'],
        capture_output=True, text=True
    )
```
