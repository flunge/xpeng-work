## 台架问题调查（广州RC）

（来源：台架问题调查 ZxaQwbba，更新时间：2026-06-02）

### 问题描述

台架运行**广州RC数据**时复现 **5080 重启**问题，同批次部分运行为整机重启，另一些仅表现为单卡渲染异常。

### 关键证据

| 信号 | 说明 |
|------|------|
| `NVRM: Xid 31 (MMU Fault)` | 驱动侧地址空间故障 |
| `CUDA out of memory` | 显存峰值 ~15838 MiB / 16303 MiB（几乎打满） |
| `CUDA illegal memory access` | 非法访存 |
| 单卡 util=0% + 另一卡高负载 | 一卡 OOM 后 continue，显存高但无有效 kernel |

### 故障链路

```
双卡各自加载大模型 → 显存长期高水位
        ↓
波峰帧/碎片化 → 分配失败或非法访存
        ↓
应用：CUDA OOM / illegal memory access
        ↓
驱动：Xid 31 (MMU Fault)
        ↓
部分时序 → 整机重启；部分时序 → 仅渲染进程异常
```

**暂定根因**：双卡高显存压力下驱动与运行时进入不稳定状态（Xid 31 + CUDA 异常），非 Linux OOM killer 路径。

### 改进方案（按优先级）

| 优先级 | 动作 |
|-------|------|
| **P0** | 统计 3DGS 模型大小（全量 2248 条），**屏蔽 >5GB 超大场景** |
| P1 | A/B：单卡运行（禁用 GPU1）对照双卡，验证 Xid 31 是否消失 |
| P1 | CUDA 连续失败阈值 → 触发全局 stop_event，避免失衡长跑 |
| P2 | 异常点打印 `torch.cuda.memory_summary()` + 即时 `nvidia-smi` |
| P2 | 分辨率/相机数/场景复杂度阶梯压测，找稳定边界 |

### 当前状态

P0 改进脚本 `stats_3dgs_size.py` 已开发，输出 avg/p95/p99（GB）及 >5GB 数量；**需在 ceph 可达环境执行**。
