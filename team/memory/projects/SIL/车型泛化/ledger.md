# 车型泛化 — 活文档

> 最后更新：2026-06-22 | 来源：memory/projects/SIL/车型泛化.md + Q2 Wiki W7–W11

## 背景与目标

A车型3DGS场景通过改变外参迁移到B/C车型，经原位渲染+Difix修图后进行开环评测，验证"跨车型趋势一致性"。最终目标支撑630多车型发版验证（GIC刘先明提出）。

核心价值：仿真场景与测试车型绑定，换车型需重新采集数据，成本极高；通过3DGS外参迁移绕过数据采集，验证趋势是否与真实数据一致。

## ref图OOD问题根因与解决方案（来源：difix_ref图模式优化 V7VMwJ6Z）

### 问题描述
换车型后，部分 3DGS 渲染图中的几何位置被 ref 图中的位置带偏，多发生于 3DGS 渲染图效果不好的区域。

### 根因分析（OOD 本质）
- **ref 与 3DGS 越像** → cross-view attention softmax 越尖锐 → 每个 3DGS token 几乎只 attend 到 ref 对应区域 token → 直接复制 ref 的几何
- **ref 与 3DGS 差很多**（如闭环仿真）→ 无锐利"对应点" → attention 弥散 → 只借鉴全局风格/纹理 → 几何不被注入
- **本质**：训练集无"高对齐 + 外参小差异"样本，attention 极尖锐行为未被正则化（分布外 OOD 问题）

### 解决方案迭代

| 版本 | 时间 | 状态 | 方案 | 效果 | 遗留问题 |
|------|------|------|------|------|---------|
| 方案1（v1） | 5/21→5/28 | Done | ref 图改为前后 ±1 秒（非时间对齐），人为制造分布差异 | 大部分 case 有改善，动态物体位置抖动问题解决 | cam3 近距离栅栏高度问题无法解决；动态障碍物车尾灯/行人动作可能丢失 |
| 方案2（v2） | ~5/29 | Done | 数据增强：训练加入换车型图作 ref（6:4 比例）+ 40% 原图 perturbation | 部分 case 改善，幻觉减轻 | 收益不均匀，部分场景仍有幻觉和跳帧 |
| 方案3（v3） | 6/1 | Done | Difix-ref 模型改造：加入车身 mask，目标车型车身不被 Diffusion 修回原车型 | 车身修复问题解决；H93/E29 RL4 组更稳 | RL2/H93 仍有错向；E29 部分指标优化后收益变弱 |

### 当前状态
- **短期最佳配置**：新模型 + -1 秒 ref 图（v1）
- **长期方案**：改造训练数据渲染链路（渲染代码改造 + 不带 mask 的训练数据重新生成）+ Difix-ref 模型加入车身 mask（v3 已完成）
- **待解决**：cam3 近距离栅栏高度（不可见新视角脑补问题）

## 验证方案设计（来源：开闭环车型验证方案 G6I4w06n）

### 整体验证框架
A 车型采集的原始场景 → 使用 B/C 等车型内外参 → 闭环原位渲染 → 开环仿真评测

核心验证逻辑（两个等价命题）：
- `1vs5 == 2vs6`：原图趋势 ≈ 同车型原位渲染趋势（验证渲染不失真）
- `3vs7 ≈ 4vs8`：原图趋势 ≈ 换车型渲染趋势（验证跨车型泛化）

数学表达：预期 $A_{nv} \to B_{nv} \approx A_{nvf} \to B_{nvf}$

其中 A/B 为模型大版本，n 为具体 Metric，v 为车型平台（α/β/γ/δ），f 为泛化版本

### 开/闭环测试场景选取

| 场景集类型 | 来源 | 规模 |
|-----------|------|------|
| 原生（PAT 固定测试集） | 各车型实采数据，通过 label 锁定防篡改 | 每车型约 20–25 个 case |
| 泛化 | 从其他车型原生场景泛化到目标车型 | 每车型约 65–75 个 case |

## CloudSim多车型Pipeline需求（来源：CloudSim闭环多车型需求 NA9IwSUG）

### 背景
当前 CloudSim 仅支持单一车型 scenario，无法直接用于不同车型的闭环仿真。需支持基于 3DGS 重渲染的多车型 scenario 生成与管理。

### 6/15 里程碑（Doing）
CloudSim 一阶段提交泛化生产 + 仿真验证，同时覆盖开环 + 闭环

### 三项关键技术改动

| 改动项 | 字段位置 | 说明 |
|--------|---------|------|
| 车型名称替换 | `vehicle_name`（顶层 + label 内） | 如 `e29_test_vehicle` → `h93aes_test_vehicle` |
| 标定数据路径替换 | `ddsDataSource.calibration` | 替换为目标车型 calibration 路径（兼容 .lz4 压缩与目录格式） |
| 3DGS calib 新增 | `3dgs_config.3dgs_calib` | 新增目标车型对应的 calib.json 路径 |

```json
"3dgs_config": {
  "oss_bucket": "cloudsim-ci-sh",
  "oss_path1": "<原始 3DGS 模型路径，保持不变>",
  "3dgs_calib": "<目标车型对应的 calib.json 路径>"
}
```

### 车型映射配置
由 3DGS 团队维护，CloudSim 侧读取。每新增支持车型需更新该配置。

| 车型名称 | calibration 路径 | calib.json 路径 |
|---------|----------------|----------------|
| h93aes_test_vehicle | oss://…/calibration.tar.lz4 | oss://…/calib.json |
| … | … | … |

### 约束
- 只允许原始 scenario 创建多车型 case，不允许多车型 case 继续转换（防止二次泛化）
- 开环/闭环统一使用同一套生产、scenario 绑定与管理逻辑

## 关键技术卡点

1. **ref 图几何干扰**：换车型后 ref 图带偏 3DGS 几何（OOD 问题）；训练数据未涵盖几何差距小的 case
2. **车身修复**：去掉 ref 图后车身被 Diffusion 修回原车型
