# 记忆系统质检规则

## 一、事实溯源（每次写入必检）

| # | 规则 | 违例 |
|---|------|------|
| 1 | 每条里程碑有精确日期（YYYY-MM-DD） | 写成"5月"而非"5/12" |
| 2 | 每条里程碑可追溯到源文档 | 从会议摘要脑补细节 |
| 3 | 量化数据（百分比/耗时/数量）必须与源文档一致 | 自己推算或四舍五入 |
| 4 | 里程碑描述使用源文档原话，不改写含义 | "大趋势一致"改写成"效果很好" |

## 二、项目边界（创建/更新项目文件时必检）

| # | 规则 | 违例 |
|---|------|------|
| 5 | `since`日期 = 该项目首次在源文档中出现 | 用Owner入职日期代替项目开始日期 |
| 6 | 一个项目文件只包含该项目的里程碑 | 把同一Owner的不同项目内容混入 |
| 7 | 项目定义与Q2 Wiki一致 | 自创Wiki里不存在的项目分类 |
| 8 | `owner`是当前实际负责人 | 已离职/调岗的人仍标为owner |

## 三、Wiki一致性（每次更新后必检）

| # | 规则 | 违例 |
|---|------|------|
| 9 | Track月目标与Q2 Wiki作战表一致 | 自己总结的月目标与Wiki不符 |
| 10 | 关键指标（复现率/效率比/产能）与Wiki最新周报一致 | 用过时的指标数据 |
| 11 | Wiki中标记为"已完成"的事项在我的文件中状态正确 | Wiki已毕业，我仍标进行中 |

## 四、人物画像（每次更新后必检）

| # | 规则 | 违例 |
|---|------|------|
| 12 | 行为观察有具体日期+出处 | "经常主动汇报"无具体例证 |
| 13 | `projects`列表与实际分配一致 | 漏掉该人物参与的项目 |
| 14 | `status`字段正确（active/on-leave/departed） | 已离职的人仍标active |

## 五、交叉索引（每次增删项目/人物时必检）

| # | 规则 | 违例 |
|---|------|------|
| 15 | 新增项目→在owner+所有contributor的projects列表中添加 | 项目文件有owner，但人物文件没该项目 |
| 16 | 人物离职→移入_departed，遍历其所有项目更新owner | 离职后项目owner悬空 |
| 17 | 项目完成→移入_archive，从相关人员projects列表移除 | 已完成项目仍在active列表 |

## 六、跨会议重复信号识别（新增规则）

| # | 规则 | 操作 | 违例 |
|---|------|------|------|
| 18 | 同一发言人在≥2次会议中提到同一阻塞/依赖 → 标记为🔴高优先级风险 | 在相关项目ledger和人物文件的风险栏写明"连续N周汇报" | 每周只提取当周内容，不与上周对比 |
| 19 | 遍历逐字稿时，按**发言人×汇报主题**做跨周归并 | 整理完所有纪要后，额外跑一遍"谁在重复说什么" | 只把每篇纪要内容分散进各项目文件就结束 |
| 20 | 核心汇报人有固定节奏（如李坤每周三）→ 优先对该发言人做跨周归并 | 对比本周三与上周三的同主题发言，标出"未解卡点" | 把逐字稿当孤立文档逐篇处理 |
| 21 | 某外部组被多次提为依赖 → 在**该组负责人的人物文件**中写入"核心阻塞依赖"节 | xiazhixun.md 应有"闭环场景集/RC路线 都在等你的metric" | 只在项目文件里提一句"待metric ready" |

**教训来源（2026-06-15 用户指出）**：李坤5/27→6/3→6/10连续三周汇报"等志勋metric ready"，全部被漏记为孤立事件。

## 七、质检流程

```
[更新前]
1. 确认要更新的是哪个项目/人物
2. 打开Q2 Wiki，确认该项目的月目标/周进展
3. 确认该项目的owner/since/contributors是否正确

[写入时]
4. 每条里程碑写日期+出处
5. 量化数据对照源文档原文

[更新后]
6. 自检：逐条过13条规则
7. 对有疑问的条目，回源文档验证
8. git commit message标注本次更新的源文档
```

### 汇报文档发布前专项检查（GIC/周报等对外文档额外执行）

```
□ ASR术语扫描：全文grep以下词，有则纠正：
    gspend | gspan | 克劳森 | 克隆界面 | cloudsion | NV fixer | clip iqa | gisplan（大小写）
□ 数字单位自查：百分比/km/台数与最新ledger一致
□ 人名一致性：姓名全程统一（不混用昵称/英文缩写/全名）
□ 红黄灯上限：单板块最多1🔴+1🟡（GIC规则）
□ 新名词有背景：首次出现的技术名词附简短定语（一句话解释是什么）
```

## 七、已知错误模式（每次更新前回顾）

| 模式 | 案例 |
|------|------|
| **同名混淆** | 同一个人（裴健宏）做两个项目（车型泛化/场景编辑），把前一个项目的内容写进后一个 |
| **时间漂移** | since日期写成Owner入职时间而非项目实际启动时间 |
| **数字偏差** | 源文档说"约200km"写成"200km"，源文档说"~79%"写成"79%" |
| **摘要替代原文** | 用AI摘要的概括替代源文档的具体描述 |
| **任务升级** | 把Q2 Wiki中一个小任务写成独立project（如场景编辑CLI）而不标注它是6/8新任务 |
| **搜索漏回复** | 用IM关键词搜索找到消息后，不读p2p对话上下文，导致遗漏老板/关键人物的回复（如搜"障碍车"找到杨星昊消息，漏掉李坤"可以尝试一下"的回复） |
| **跳过嵌套文档** | Q2 Wiki嵌套文档列表里看到核心文档（如OKR文档LpbTdfU95o），凭标题判断优先级后跳过，导致项目边界/Owner信息缺失 |
| **段落混写** | 把目标/进展/风险/计划揉成一段话，而不是分bullet写 |
| **不跑质检** | 创建了quality-rules.md但从不在写文档时使用 |
| **周报写历史** | 在周报进展段回顾前几个月的事。周报只写本周delta，历史放project文件 |
| **多进展不拆子bullet** | 一个项目有多条进展时揉成一段话。应该用子bullet逐条列出 |
| **内部黑话/过度省略** | "保质量""操作agent化""卡IT部署""在搞"——老板看不懂。写成完整判断+具体行动 |
| **逐字稿原文照搬** | "克隆界面""克劳森"直接搬进周报。应转换为正式名称（Cloudsim） |
| **ASR语音识别错误** | gspend/gspan → gsplat；其他常见ASR错字见下方「ASR术语纠错表」。汇报文档发布前必须过一遍 |

## 八、ASR术语纠错表（汇报文档发布前必检）

每次写入 ledger 或汇报文档时，先对照此表将 ASR 错字替换为正式名称。遇到不认识的新词先问用户，禁止原样写入。

| ASR 识别错误 | 正确术语 | 说明 |
|---|---|---|
| gspend、gspan | gsplat | 3DGS渲染库，语音识别把"gsplat"识别成相近音 |
| 克劳森、cloudsion | Cloudsim | 仿真平台名称 |
| 克隆界面 | Cloudsim | 同上 |
| difix、DI fix | Difix | Diffusion Fixer旧架构名称 |
| NV fixer、NV-fixer | NVFixer | Diffusion Fixer新架构名称（NV = NeuroVision/新架构代号） |
| clip IQA、CLIP iqa | CLIP-IQA | 图像质量评测模块 |
| gisplan、G plan | Gisplan | 仿真渲染pipeline名称 |
| UCP | UCP | 正确（勿改，是subrun的upstream control plane缩写） |

**补充规则**：
- 遇到任何「读音像英文缩写但拼写奇怪」的词，先标记 `[ASR?]` 并在当次会话结束前向用户确认
- 确认后补充到此表，下次自动纠正

## 九、汇报/述职叙事规则（2026-06-26 用户指出）

**核心教训**：让你参照某个框架（价值观/干部七行/能力模型）写汇报时，框架要**隐含在内容表述里**，绝不能写成"对照「X」逐条看：- 客户至上：…"这种贴标签式列表——没有人会这样汇报。

| # | 规则 | 违例 | 正解 |
|---|------|------|------|
| 22 | 框架（价值观/七行/STAR）只做**暗线**，用做事的方式自然体现，不出现"对照X""逐条看"字样 | "**客户至上**：把仿真做成业务可用" | "判断先做什么，我看的是发版 gating 卡不卡、客诉能不能快点复现——而不是 demo 好不好看" |
| 23 | 用 **STAR** 讲事：情境痛点→我的判断与行动→量化结果，别堆流水账/苦劳 | "改了N个bug、开了N次会" | "闭环场景集卡在 metric，我没干等上游，而是…，结果…" |
| 24 | 体现 **Ownership**：讲清问题背后的根因和你从源头做的取舍，不是罗列动作 | "做了漏斗看板" | "数据一直在丢但说不清丢在哪，我建漏斗看板把每个丢失点标出来" |
| 25 | **去 AI/学生气**：少用"赋能/抓手/闭环打通"等套话和工整排比；用人话、口语化、有判断有取舍 | "用数据引擎放大团队效率" | "一个组干不成，得拉着业务/硬件/评估一起" |
| 26 | 成果**对齐业务价值**，不止技术描述 | "训练提速2.6×" | "训练提速2.6×，客诉来了能当天复现" |
| 27 | 改写汇报时**不动既定基本结构**（章节/小节顺序），只改表述与衔接 | 重排章节 | 保持五章结构，只润色语言 |

> 经验来源：STAR 全景模型（大厂转正答辩评审逻辑）+ 用户 2026-06-26 反馈"表述太AI、框架别贴标签"。
