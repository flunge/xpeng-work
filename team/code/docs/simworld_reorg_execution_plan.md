# simworld 仓库重构与回归测试执行方案

## 1. 为什么做这次重构，以及预期收益

### 1.1 为什么现在必须做

- 仓库已从“单一模型研发”演进到“预处理 + 多模型 + 生产 + 闭环仿真 + HIL”的复合工程，历史结构难以支撑持续迭代。
- 关键入口分散且命名不统一，导致新需求落地成本高、问题定位慢、交接依赖个人经验。
- 旧链路与新链路并存，重构窗口期如果不统一治理，后续会持续放大维护成本与线上风险。

### 1.2 预期收益（业务和研发层面）

1. **研发效率提升**
   - 四大主链路入口清晰，减少“找脚本、找配置、找日志”的无效时间。
   - 模型目录按“一模型一目录”管理，降低跨目录跳转和隐式依赖成本。

2. **质量与稳定性提升**
   - 回归测试按 Fuyao/UCP/仿真/HIL 四域固化，降低改动引入隐性回归的概率。
   - 废弃入口有明确 `deprecated` 与替代路径，减少误用旧脚本导致的故障。

3. **协作与可维护性提升**
   - 新人可按目录快速理解系统边界，缩短上手周期。
   - 文档与代码入口一致，跨团队协作（算法/仿真/生产）沟通成本下降。

4. **可持续演进能力提升**
   - 目录分层后可支持后续模块继续扩展（新模型、新链路）而不破坏整体结构。
   - 每次重构都有迁移映射、回滚点和验收标准，变更治理更可控。

## 3. 目录映射与迁移原则

- `pipeline/fuyao` -> `pipeline/fuyao`
- `pipeline/ucp` -> `pipeline/ucp`
- `hil` -> `hil`
- `xpeng_raster` -> `libs/xpeng_raster`
- `agents` -> `agents`
- `dockerfile` 并入 `tools/dockerfile`
- `sim_interface` 保持为一级目录，`render_sim` 系列继续保留在 `omnire_joint_trainning/src/reconic`
- `models` 下按模型一目录管理，作为后续代码归并的主入口

原则：

1. 先完成结构迁移与引用修复，再做代码清理。
2. 所有迁移动作必须保留短期兼容（脚本入口别名或清晰迁移说明）。
3. 以 Fuyao/UCP/仿真/HIL 四条链路可回归为验收基线。

## 4. 当前仓库问题

### 4.1 结构与职责问题

- 生产、训练、仿真、工具脚本分布在多个目录，职责边界不清晰，新人很难快速定位入口。
- 历史路径与现有主链路并存（例如 UCP 旧入口与新入口并存），导致“看起来可用、实际已废弃”的误用风险。
- 同类能力有重复目录或重复脚本（尤其 debug、部署衍生脚本），维护成本高。

### 4.2 代码与配置问题

- 训练/渲染/预处理脚本命名和参数风格不统一，迁移和排障需要额外上下文。
- 模型相关代码未按“一个模型一个目录”清晰归档，difix/nvfixer/street_gaussians/g3r 等协同关系不直观。
- 历史配置与实验配置缺乏分层管理，容易误用过期配置。

### 4.3 交付与运维问题

- 回归链路未形成统一 Runbook，跨人协作时依赖个人经验。
- 变更影响面评估和回滚点记录不统一，重构期间风险难量化。
- 仓库文档与真实入口存在偏差，降低后续维护效率。

## 5. 重构后可解决项与判断标准

### 5.1 预期可解决问题

- 目录按职责收敛后，四大核心链路（Fuyao/UCP/闭环仿真/HIL）入口统一、可检索。
- 模型代码在 `models/` 下“一模型一目录”，减少跨目录跳转和隐式依赖。
- 部署链路收敛到 `pipeline/fuyao` 与 `pipeline/ucp`，脚本维护面明显缩小。
- 基础库、工具、Agent 目录职责独立（`libs`、`tools`、`agents`），降低改动互相污染。
- 通过回归计划与 Runbook 固化，减少“只能由熟悉历史的人维护”的风险。

### 5.2 判断是否“解决”的验收标准

1. **结构清晰度**
   - 新人在 30 分钟内可定位四条主链路入口并跑通至少 1 条 smoke。
   - 每个核心目录都有 README 或 Runbook，说明“用途、入口、依赖、维护人”。

2. **链路可用性**
   - 4 大回归域全部通过（Fuyao、UCP/IPS、difix/nvfixer、闭环仿真）。
   - 关键链路失败时可在 15 分钟内定位到入口脚本和关键日志路径。

3. **维护成本**
   - 废弃入口有明确 `deprecated` 标识和替代路径。
   - 重复脚本和无引用配置完成清理并有变更记录。

4. **交付质量**
   - 输出完整新框架维护文档，覆盖目录规范、开发流程、发布与回归清单。
   - 每次结构调整有可回滚方案，并在文档中保留回滚指引。

## 6. 实施计划

### 6.1 文件架构重构（0.5 天）

- 新建目标目录骨架（`pipeline`/`models`/`libs`/`agents` 等）。
- 迁移目录并修复显式路径引用（bash、python、README、配置）。
- 更新根 README 的真实入口与路径说明。
- 输出迁移清单：旧路径、 新路径、兼容策略。

### 6.2 3DGS 训练/渲染/预处理代码整理（1 天）

- Reconic 训练与渲染入口脚本归一（`deploy_cmd`、`fuyao` 脚本）。
- 预处理链路脚本归一（`xpeng_data_process` + `pipeline/fuyao`）。
- 删除确定废弃的重复脚本与无效配置，保留变更记录。
- 输出《训练与预处理脚本映射表》。

### 6.3 闭环仿真与 UCP 生产代码整理（1 天）

- UCP 入口脚本梳理（clip/subrun、多车型、difix 分支）。
- 闭环仿真入口梳理（`sim_interface` + `reconic/simulator`）。
- 对历史脚本增加 `deprecated` 标识与替代入口说明。
- 输出《生产与仿真入口 Runbook》。

## 7. 回归测试计划

### 7.1 Fuyao 链路（预处理 + 3DGS 训练 + 3DGS 渲染，2 天）

- 整理并验证脚本：
  - 预处理提交脚本
  - 训练提交脚本
  - 渲染提交脚本
- 每条链路至少执行 1 个 smoke case + 1 个标准 case。
- 验收指标：任务成功率、关键产物落盘、核心时延无异常回退。

### 7.2 UCP/IPS 生产链路（多车型 + clip/subrun，2 天）

- 覆盖场景：
  - 多车型生产
  - clip 生产
  - subrun 生产
- 校验项：入口参数兼容、OSS 产物结构、失败重试逻辑。
- 验收指标：三类场景均可全流程跑通。

### 7.3 difix/nvfixer 链路（训练 + infer，1 天）

- 整理 Fuyao 训练脚本与推理脚本。
- 验证模型产出、推理结果和主流程集成点。
- 验收指标：训练与 infer 各至少 1 条链路可独立跑通。

### 7.4 闭环仿真链路（2 天）

- 校验 `sim_interface` 接口调用一致性。
- 验证 reconic 仿真渲染主流程。
- 验证关键后处理与可视化产物。
- 验收指标：闭环仿真链路可稳定复现。

## 8. 代码清理方法

- 方式：AI 初筛 + 人工复核双层清理。
- AI 负责：重复代码、无引用脚本、旧配置模板、命名不一致检测。
- 人工负责：功能边界确认、线上兼容判断、删除审批。
- 要求：每次删除或迁移必须有可追溯记录（原因、影响范围、回滚方式）。

## 9. 交付物

1. 新目录结构与迁移映射文档。
2. 四大核心链路 Runbook（Fuyao、UCP、闭环仿真、HIL）。
3. 回归测试报告（按 4 个测试域输出）。
4. 代码清理记录（删除清单、保留原因、风险说明）。
5. 新框架维护文档（面向后续开发者更新维护）。

## 10. 风险与保障

- 风险 1：迁移后隐式路径失效。  
  保障：全仓路径扫描 + 冒烟脚本前置执行。
- 风险 2：旧脚本仍被线上任务引用。  
  保障：先加 `deprecated` 与兼容跳转，再分批下线。
- 风险 3：多链路并行导致回归周期拉长。  
  保障：按优先级并行排期，先保生产主链路。

## 11. 建议排期（合计 9.5 天）

- 架构重构：0.5 天
- 训练/渲染/预处理整理：1 天
- 闭环仿真/UCP 整理：1 天
- Fuyao 回归：2 天
- UCP/IPS 回归：2 天
- difix/nvfixer 回归：1 天
- 闭环仿真回归：2 天

> 建议每日同步：进展、阻塞、风险、次日计划；每阶段输出可回滚点。
